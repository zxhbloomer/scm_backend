# AI聊天思考过程展示 - 设计方案

## 1. 需求概述

在SCM智能助手的聊天中，当用户发送问题触发工作流执行时，用户当前只能看到"SCM正在思考..."的加载动画，无法了解AI正在做什么。

本方案在工作流执行期间，向用户展示**实时的思考步骤列表**（节点级粒度），完成后自动消失，通过底部标签可回看执行详情。

## 2. 参考项目

| 项目 | 方案特点 | 评估 |
|------|---------|------|
| FastGPT | 16种SSE事件、`flowNodeStatus`实时状态、`showStatus`守卫过滤 | 功能全面，事件机制值得借鉴 |
| Spring AI Alibaba DeepResearch | `GraphLifecycleListener`回调、`Sinks.Many`侧信道、ThoughtChain组件 | 框架原生支持，架构参考 |
| Spring AI | Flux响应式流、`ServerSentEvent`传输、Advisor链式拦截 | 底层框架参考 |

### 2.2 调研关键发现

#### Spring AI Alibaba Graph框架原生支持

框架内置`GraphLifecycleListener`接口，提供节点执行前后的回调钩子：

```java
public interface GraphLifecycleListener {
    // 节点执行前 → 对应 node_start 事件
    default void before(String nodeId, Map<String, Object> state, RunnableConfig config, Long curTime) {}
    // 节点执行后 → 对应 node_complete 事件
    default void after(String nodeId, Map<String, Object> state, RunnableConfig config, Long curTime) {}
    default void onStart(String nodeId, Map<String, Object> state, RunnableConfig config) {}
    default void onComplete(String nodeId, Map<String, Object> state, RunnableConfig config) {}
    default void onError(String nodeId, Map<String, Object> state, Throwable ex, RunnableConfig config) {}
}
```

注册方式：通过`CompileConfig`在`compile()`时传入。SCM当前使用无参`compile()`，需改为：

```java
CompileConfig config = CompileConfig.builder()
    .lifecycleListeners(List.of(nodeEventListener))
    .build();
app = mainStateGraph.compile(config);
```

#### 事件发送机制：Sinks.Many侧信道

`GraphLifecycleListener`回调是同步`void`方法，无法直接往Flux管道推事件。采用Spring AI Alibaba DeepResearch相同的`Sinks.Many`模式：

```java
// WorkflowEngine中
private Sinks.Many<WorkflowEventVo> eventSink = Sinks.many().unicast().onBackpressureBuffer();

// Listener回调中
eventSink.tryEmitNext(WorkflowEventVo.createNodeStartData(...));

// executeWorkflow()中合并两路Flux
return Flux.merge(
    graphStream.flatMap(this::handleGraphResponse),  // 原有output/chunk事件
    eventSink.asFlux()                                // 新增node_start/complete事件
);
```

#### FastGPT的简洁模式

FastGPT只发`flowNodeStatus {status:'running', name:'节点名'}`一种事件，靠下一个节点的running隐式标记上一个节点完成。我们选择显式的`node_start` + `node_complete`两事件模式，语义更清晰，前端不用猜。

## 3. 交互设计

### 3.1 三阶段时序

#### 阶段1：执行中 — 步骤列表实时展开

```
┌─ AI 消息气泡 ────────────────────────────┐
│                                          │
│  ✓ 意图识别     采购咨询           0.3s  │
│  ✓ 知识库检索   匹配3条            1.2s  │
│  ⏳ 生成回答    正在生成...               │
│                                          │
│  ─────────────────────────────────────── │
│  采购订单的创建步骤如下...                │
│                                          │
└──────────────────────────────────────────┘
```

- 每个节点开始执行时，新增一行 `⏳ {节点名称} {动作}...`
- 节点完成后，变为 `✓ {节点名称} {摘要} {耗时}`
- LLM流式文本在步骤列表下方同步输出（分割线隔开）

#### 阶段2：完成 — 步骤列表直接消失

```
┌─ AI 消息气泡 ────────────────────────────┐
│                                          │
│  采购订单的创建步骤如下：                  │
│  1. 进入采购管理模块                      │
│  2. 点击新建按钮，填写供应商...            │
│  3. 提交审批                              │
│                                          │
│  ┌─────────┐ ┌───────┐ ┌─────────┐     │
│  │📚 引用 3条│ │⏱ 2.1s │ │📄 查看详情│     │
│  └─────────┘ └───────┘ └─────────┘     │
└──────────────────────────────────────────┘
```

- 工作流完成（收到isComplete=true）后，步骤列表立即消失
- 底部出现元信息标签行

#### 阶段3：回看 — 点击"查看详情"展开

```
┌─ AI 消息气泡 ────────────────────────────┐
│                                          │
│  采购订单的创建步骤如下：...               │
│                                          │
│  ┌─ 执行详情 ────────────────────────┐   │
│  │                                    │   │
│  │  ✓ 意图识别     采购咨询     0.3s  │   │
│  │                                    │   │
│  │  ▶ 知识库检索   匹配3条      1.2s  │   │
│  │  ┌───────────────────────────┐    │   │
│  │  │ 📄 采购流程规范.pdf          │    │   │
│  │  │    采购订单审批流程需要...    │    │   │
│  │  │ 📄 供应商管理手册.docx       │    │   │
│  │  │    供应商评级标准分为...      │    │   │
│  │  └───────────────────────────┘    │   │
│  │                                    │   │
│  │  ✓ LLM回答     deepseek     0.6s  │   │
│  │                                    │   │
│  └────────────────────────────────────┘   │
│                                          │
│  ┌─────────┐ ┌───────┐ ┌─────────┐     │
│  │📚 引用 3条│ │⏱ 2.1s │ │📄 收起详情│     │
│  └─────────┘ └───────┘ └─────────┘     │
└──────────────────────────────────────────┘
```

- "查看详情"变为"收起详情"
- 每个节点行可点击展开查看具体内容

### 3.2 展示的节点类型（6种）

| 节点类型 | 执行中文本 | 完成摘要 | 展开详情内容 |
|---------|-----------|---------|------------|
| 分类器 | `⏳ 意图识别 分析中...` | `✓ 意图识别 → 采购咨询` | 无需展开 |
| 知识库检索 | `⏳ 知识库检索 检索中...` | `✓ 知识库检索 → 匹配N条` | 每条：文档名 + 内容摘要 |
| 临时知识库 | `⏳ 创建临时知识库...` | `✓ 临时知识库 → 创建完成` | 无需展开 |
| LLM回答 | `⏳ 生成回答 正在生成...` | `✓ 生成回答 → {模型名}` | Token用量 |
| MCP工具 | `⏳ 调用工具 {工具名}...` | `✓ 调用工具 → {工具名}` | 参数JSON + 结果JSON |
| 文档解析 | `⏳ 解析文档...` | `✓ 文档解析 → 提取完成` | 无需展开 |

**不展示的节点**：Start、End、Template、Switcher、SubWorkflow、HttpRequest、MailSend、KeywordExtractor、FaqExtractor

### 3.3 底部标签规则

| 标签 | 出现条件 | 内容 |
|------|---------|------|
| 📚 引用 N条 | 工作流含知识库检索且有结果 | 点击滚动到详情中的知识库部分 |
| 🔧 工具 N个 | 工作流含MCP工具节点 | 点击滚动到详情中的工具部分 |
| ⏱ X.Xs | 始终显示（工作流模式） | 总耗时 |
| 📄 查看详情 | 始终显示（工作流模式） | 切换展开/收起执行详情面板 |

### 3.4 特殊场景

| 场景 | 处理 |
|------|------|
| 普通聊天（无工作流） | 不显示思考过程，保持现有"SCM正在思考..."行为 |
| 工作流只有1个LLM节点 | 只显示 `⏳ 生成回答 正在生成...`，完成后消失 |
| 节点执行失败 | 显示 `✗ {节点名} 失败`（红色） |
| 并行节点 | 按完成顺序依次出现 |

## 4. 数据结构设计

### 4.1 新增后端SSE事件（2种）

在现有4种事件（runtime、chunk、output、interrupt）基础上，新增：

#### node_start事件

节点开始执行时发送。

```json
{
  "type": "node_start",
  "node": "uuid-xxx",
  "nodeName": "KnowledgeRetrieval",
  "nodeTitle": "知识库检索",
  "timestamp": 1709280000000
}
```

#### node_complete事件

节点执行完成时发送。

```json
{
  "type": "node_complete",
  "node": "uuid-xxx",
  "nodeName": "KnowledgeRetrieval",
  "nodeTitle": "知识库检索",
  "duration": 1200,
  "summary": {
    "matchCount": 3,
    "items": [
      { "docName": "采购流程规范.pdf", "snippet": "采购订单审批流程需要经过..." },
      { "docName": "供应商管理手册.docx", "snippet": "供应商评级标准分为..." }
    ]
  }
}
```

各节点类型的summary结构：

| 节点类型 | summary字段 |
|---------|------------|
| 分类器 | `{ "result": "采购咨询" }` |
| 知识库检索 | `{ "matchCount": 3, "items": [{ "docName", "snippet" }] }` |
| 临时知识库 | `{ "kbUuid": "..." }` |
| LLM回答 | `{ "model": "deepseek-chat", "promptTokens": 500, "completionTokens": 200 }` |
| MCP工具 | `{ "toolName": "openPage", "params": {...}, "result": {...} }` |
| 文档解析 | `{ "fileCount": 1 }` |

### 4.2 前端消息数据结构扩展


## 5. 后端技术方案

### 5.1 架构总览

```
GraphLifecycleListener.before()  ──→  Sinks.Many<WorkflowEventVo>  ──→  Flux.merge()  ──→  SSE
GraphLifecycleListener.after()   ──→           (侧信道)              ──→      ↑
                                                                            ↑
graphResponseStream()  ──→  handleGraphResponse()  ──→  原有output/chunk事件 ──┘
```

### 5.2 实现要点

#### 5.2.1 WorkflowEngine改动

1. 新增`Sinks.Many<WorkflowEventVo> eventSink`字段
2. 实现`GraphLifecycleListener`内部类，在`before()/after()`中发送事件
3. `compile()`时通过`CompileConfig`注册listener
4. `executeWorkflow()`中`Flux.merge()`合并两路事件流
5. 可见节点过滤：listener内部根据`nodeName`判断是否属于6种可见类型

#### 5.2.2 WorkflowEventVo新增工厂方法

- `createNodeStartData(nodeUuid, nodeName, nodeTitle, timestamp)`
- `createNodeCompleteData(nodeUuid, nodeName, nodeTitle, duration, summary)`

#### 5.2.3 WorkflowEventAdapter适配

- `node_start` → `ChatResponseVo`增加`nodeStatus`相关字段
- `node_complete` → `ChatResponseVo`增加`nodeStatus`+`nodeSummary`字段

#### 5.2.4 各节点summary数据提取

通过`after()`回调的`state`参数获取节点输出数据，构造summary。

## 6. 分步实施计划

> 核心策略：**先用1种节点打通全链路验证可行性，再逐步扩展**

### Step 1：单节点全链路验证（知识库检索）

**目标**：选择"知识库检索"节点，打通 后端事件发送 → SSE传输 → 前端步骤列表 的完整链路

**选择知识库检索的理由**：
- 出现频率最高（几乎每个业务工作流都有）
- 非流式同步节点，有明确的开始/结束边界
- summary有代表性（matchCount + items）
- 一个节点即可验证整条管道

**后端改动**：
| 文件 | 改动 |
|------|------|
| WorkflowEventVo.java | 新增`createNodeStartData()`和`createNodeCompleteData()`工厂方法 |
| WorkflowEngine.java | 新增Sinks.Many + GraphLifecycleListener + CompileConfig注册 + Flux.merge() |
| WorkflowEventAdapter.java | 新事件类型透传为ChatResponseVo |
| ChatResponseVo.java | 新增nodeStatus相关字段 |

**前端改动**：
| 文件 | 改动 |
|------|------|
| aiChatService.js | SSE解析新增node_start/node_complete事件处理 |
| chat.js (store) | 消息数据结构扩展workflowProcess + mutations |
| MessageList.vue | 集成ThinkingSteps组件 |
| 新建: ThinkingSteps.vue | 步骤列表组件（⏳ running / ✓ completed / ✗ error） |

**Listener过滤规则**：Step 1中只对`KnowledgeRetrieval`节点发送事件，其他节点静默跳过

**summary**：Step 1中`node_complete`的summary包含`matchCount`，暂不传`items`详情

**验证方式**：
1. F12 Network → EventStream，确认SSE流中出现node_start/node_complete事件
2. 聊天气泡内出现 `⏳ 知识库检索 检索中...` → `✓ 知识库检索 匹配N条 X.Xs`
3. 工作流完成后步骤列表消失

### Step 2：扩展到其余5种节点

**目标**：补全6种可见节点的事件发送和前端展示

**改动范围**：
| 节点类型 | 新增summary数据 | 前端展示文本 |
|---------|----------------|------------|
| 分类器 | `{ result }` | ✓ 意图识别 → 采购咨询 |
| 临时知识库 | `{ kbUuid }` | ✓ 临时知识库 → 创建完成 |
| LLM回答 | `{ model }` | ✓ 生成回答 → deepseek |
| MCP工具 | `{ toolName }` | ✓ 调用工具 → openPage |
| 文档解析 | `{ fileCount }` | ✓ 文档解析 → 提取完成 |

**Listener过滤规则**：扩展白名单为全部6种可见节点

**验证方式**：覆盖不同类型工作流，确认各节点步骤均正确显示

### Step 3：底部标签 + 详情面板（完善体验）

**目标**：完成阶段2（底部标签）和阶段3（回看详情）的交互

**改动范围**：
| 文件 | 改动 |
|------|------|
| 6种节点类 | summary补充完整详情数据（items、params等） |
| 新建: ResponseTags.vue | 底部标签组件（引用N条、工具N个、耗时、查看详情） |
| 新建: ExecutionDetail.vue | 执行详情面板组件（展开/收起 + 节点内容详情） |

**验证方式**：完整的三阶段交互流程（执行中→消失→回看）

## 7. 不做的事情（KISS原则）

- 不做节点连线图可视化
- 不做实时Token计数动画
- 不做并行节点的分支展示
- 不做子工作流递归展开
- 不做节点日志实时滚动
