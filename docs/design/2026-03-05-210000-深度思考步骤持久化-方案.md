# 深度思考步骤持久化-方案

## 1. 问题背景

工作流执行过程中产生的"深度思考步骤"（node_start/node_complete事件）通过SSE实时推送给前端展示（如"意图分类 → 知识库检索 → 美化返回"等步骤），但这些数据仅存在于前端Vuex内存中。用户刷新页面后，`loadMessages()`从数据库加载历史消息，由于`ai_conversation_content`表没有`workflow_steps`字段，思考步骤全部丢失。

**期望行为**：刷新页面后，历史消息的思考步骤仍然可见（折叠状态），用户可以展开查看。

## 2. 完整调用链路

### 2.1 写入链路（当前）
```
用户发送消息
    |
ChatPanel.handleSendMessage() → emit('send-message')
    |
ChatBubble → store.dispatch('chat/sendMessage')
    |
aiChatService.sendMessageStream() → POST /api/v1/ai/conversation/chat/stream (SSE)
    |
AiConversationController.chatStream()
    |
workflowRoutingService.routeAndExecute()
    |
WorkflowEngine.executeWorkflow()
    |
    ├── node_start/node_complete事件 → SSE → 前端SET_WORKFLOW_PROCESS_NODE
    |
    └── doOnComplete → updateWorkflowComplete()
    |
AiConversationController.handleResponse() [isComplete=true]
    |
    ├── aiConversationContentService.saveContent(user消息)
    └── aiConversationContentService.saveContent(ai消息, runtimeUuid, aiOpenDialogPara)
         ↑ 此处缺少workflow_steps保存
```

### 2.2 读取链路（当前）
```
页面加载/刷新
    |
store.dispatch('chat/loadMessages')
    |
aiChatService.getAiChatDetail(conversationId)
    |
GET /api/v1/ai/conversation/chat/list/{conversationId}
    |
ExtAiConversationContentMapper.selectByConversationId()
    |
返回消息列表（无workflow_steps字段）
    |
前端formattedMessages → 无workflowSteps属性 → 思考步骤不显示
```

### 2.3 修改后的链路
```
写入：handleResponse() → saveContent()时额外传入workflowStepsJson
      workflowStepsJson来源：根据runtimeId查询ai_conversation_runtime_node + ai_workflow_node
      重建为前端steps格式JSON → 存入ai_conversation_content.workflow_steps

读取：selectByConversationId() SQL增加workflow_steps字段
      前端loadMessages()中映射msg.workflow_steps → message.workflowSteps
```

## 3. 问题诊断

| 问题 | 当前 | 期望 |
|---|---|---|
| 工作流执行中的思考步骤 | 实时SSE推送，前端显示 | 不变 |
| 刷新页面后的思考步骤 | 消失 | 从数据库恢复并显示 |
| 非工作流消息 | 无思考步骤 | 不变（workflow_steps为null） |

**(B) 新增`buildWorkflowStepsJson`方法**：

根据runtimeId查询`ai_conversation_runtime_node` + `ai_workflow_node`表，重建前端格式的steps JSON。

```java
/**
 * 根据运行时ID构建工作流思考步骤JSON
 * 从ai_conversation_runtime_node和ai_workflow_node表重建前端steps格式
 *
 * @param runtimeId 运行时ID（ai_conversation_runtime.id）
 * @return JSON字符串，如果无节点数据返回null
 */
public String buildWorkflowStepsJson(Long runtimeId) {
    if (runtimeId == null) return null;

    List<AiConversationRuntimeNodeVo> nodes = conversationRuntimeNodeService.listByWfRuntimeId(runtimeId);
    if (nodes == null || nodes.isEmpty()) return null;

    JSONArray stepsArray = new JSONArray();
    for (AiConversationRuntimeNodeVo node : nodes) {
        JSONObject step = new JSONObject();
        step.put("nodeUuid", node.getRuntimeNodeUuid());

        // 从ai_workflow_node获取组件名称
        String componentName = workflowNodeService.getComponentNameByNodeId(node.getNodeId());
        step.put("nodeName", componentName);
        step.put("nodeTitle", node.getNodeTitle());
        step.put("status", node.getStatus() == 3 ? "done" : "running");

        // 耗时：u_time - c_time（毫秒）
        long duration = 0;
        if (node.getC_time() != null && node.getU_time() != null) {
            duration = java.time.Duration.between(node.getC_time(), node.getU_time()).toMillis();
        }
        step.put("duration", duration);

        stepsArray.add(step);
    }

    return stepsArray.toJSONString();
}
```

## 5. KISS原则7问回答

1. **真问题？** 是。用户明确反馈刷新后思考步骤消失，影响信息回溯体验。
2. **更简方法？** 已是最简——1个ALTER TABLE + 写入时多存一个JSON字段 + 读取时多返回一个字段。
3. **会破坏什么？** 无破坏。新增字段默认null，旧数据不受影响；saveContent增加参数但调用方同步修改。
4. **真需要？** 是。用户希望历史对话完整保留，包括思考过程。
5. **过度设计？** 无。直接存JSON，避免了运行时动态重建的复杂度。
6. **话题模糊？** 不模糊。数据来源（`ai_conversation_runtime_node`）、存储位置（`ai_conversation_content.workflow_steps`）、前端读取（`msg.workflow_steps` → `workflowSteps`）全部明确。
7. **代码实施注意事项？** 是。更新使用bean操作（selectById→set→updateById）；查询使用SQL（mapper中@Select注解）；Entity字段用驼峰+@TableField映射；VO字段用下划线。

## 7. 向后兼容分析

- 新增`workflow_steps`字段默认NULL → 旧数据无影响
- 非工作流消息（纯LLM对话）→ `runtimeId`为null → `workflowSteps`为null → 不显示
- 前端`getNodeSteps()`已实现优先读实时数据、次读持久化数据的逻辑（MessageList.vue:536-546）
- 前端`isStreamComplete()`通过检查实时数据是否存在判断流状态（MessageList.vue:561-564）

## 8. 风险分析

| 风险 | 等级 | 说明 |
|---|---|---|
| workflow_steps JSON过大 | 极低 | 每条消息最多十几个节点，JSON不超过2KB。LONGTEXT足够 |
| 数据冗余（runtime_node表已有） | 低 | 冗余换取查询简洁，且前端格式已预处理，加载时无需关联查询 |
| saveContent签名变化 | 低 | 所有调用方在同一Controller内，同步修改 |
| ALTER TABLE锁表 | 极低 | LONGTEXT字段加NULL默认值，MySQL 8.x支持instant DDL |
