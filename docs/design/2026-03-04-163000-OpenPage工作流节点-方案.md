# OpenPage工作流节点设计方案

## 1. 需求背景

工作流"采购-项目管理-新增"(workflow 34)执行完成后，End节点直接输出原始JSON到聊天窗口。用户看到一堆JSON数据毫无意义。

**期望**：
1. 聊天窗口显示友好自然语言（如"已为您准备好信息，页面即将打开"）
2. 同时自动打开业务弹窗，预填JSON中的数据
3. 后续前端可渲染"打开页面"按钮（前端方案不在本次范围）

**方案**：新增"打开前端页面"(OpenPage)工作流节点类型。

## 2. 完整调用链路

```
前端ChatPanel → WorkflowRoutingService.executeWorkflowByUuid()
                     ↓
              WorkflowStarter.streaming()
                     ↓
              WorkflowEngine.executeWorkflow()
                     ↓
       ┌─────────────────────────────────────┐
       │  大模型节点(187) → OpenPage节点(新)  │
       │       → End节点(194)                 │
       └─────────────────────────────────────┘
                     ↓
              WorkflowEventVo事件流
              (通过Flux.merge合并graphEventFlux + nodeEventFlux)
                     ↓
              WorkflowRoutingService.executeWorkflowByUuid()中
              eventFlux.map() → 拦截workflow_output_data事件
                     ↓
              Flux.concat(contentFlux, 完成事件)
              完成事件中携带workflowOutputData字段
                     ↓
              前端ChatPanel接收
```

**两条数据通道**：
- **主通道**（已有）：OpenPage调LLM → 友好文字 → chunk事件 → End → 聊天显示
- **侧通道**（新增）：OpenPage → `wfState.workflowOutputData` → WorkflowEngine.doOnComplete → sink发送workflow_output_data事件 → WorkflowRoutingService拦截并存储 → 完成事件携带 → 前端弹窗

### 4.2 新建：OpenPageNode.java

**路径**：`scm-ai/src/main/java/com/xinyirun/scm/ai/workflow/node/openpage/OpenPageNode.java`

```java
package com.xinyirun.scm.ai.workflow.node.openpage;

import com.xinyirun.scm.ai.bean.entity.workflow.AiWorkflowComponentEntity;
import com.xinyirun.scm.ai.bean.vo.workflow.AiWorkflowNodeVo;
import com.xinyirun.scm.ai.workflow.NodeProcessResult;
import com.xinyirun.scm.ai.workflow.WfNodeState;
import com.xinyirun.scm.ai.workflow.WfState;
import com.xinyirun.scm.ai.workflow.WorkflowUtil;
import com.xinyirun.scm.ai.workflow.node.AbstractWfNode;
import lombok.extern.slf4j.Slf4j;

/**
 * 打开前端页面节点
 *
 * 将上游大模型输出的JSON数据存入侧通道(wfState.workflowOutputData)，
 * 同时调用LLM生成友好的自然语言回复显示在聊天窗口。
 */
@Slf4j
public class OpenPageNode extends AbstractWfNode {

    private static final String PROMPT_TEMPLATE =
        "你是SCM供应链管理系统的智能助手。用户请求了一个业务操作，系统已准备好操作数据并将自动打开对应的业务页面。\n\n" +
        "请根据以下JSON数据，用简洁友好的中文回复用户：\n" +
        "1. 告知操作已准备就绪，页面即将打开\n" +
        "2. 简要列出JSON中的关键业务字段（如项目名称、金额等），忽略系统字段（如ai_new_route、id等）\n" +
        "3. 提醒用户在打开的表单中确认并补充信息\n\n" +
        "不要输出JSON本身，不要输出Markdown代码块，只输出自然语言。控制在100字以内。\n\n" +
        "JSON数据：\n";

    public OpenPageNode(AiWorkflowComponentEntity wfComponent, AiWorkflowNodeVo nodeDef,
                        WfState wfState, WfNodeState nodeState) {
        super(wfComponent, nodeDef, wfState, nodeState);
    }

    @Override
    public NodeProcessResult onProcess() {
        OpenPageNodeConfig nodeConfig = checkAndGetConfig(OpenPageNodeConfig.class);
        String inputText = getFirstInputText();
        log.info("OpenPage节点开始处理，输入长度: {}", inputText != null ? inputText.length() : 0);

        // 1. 将原始数据存入侧通道，工作流完成时通过事件传递给前端
        wfState.setWorkflowOutputData(inputText);
        log.info("OpenPage节点已将数据存入workflowOutputData侧通道");

        // 2. 调用LLM生成友好的自然语言回复
        String prompt = PROMPT_TEMPLATE + inputText;
        WorkflowUtil.streamingInvokeLLM(wfState, state, node, nodeConfig.getModelName(), prompt);

        return new NodeProcessResult();
    }
}
```

核心逻辑30行以内，职责单一：存JSON + 调LLM。完全复用LLMAnswerNode的模式。

## 5. KISS原则7问题回答

1. **这是真问题？** ✅ 是。用户看到原始JSON = 体验崩坏，真实存在于所有使用AI工作流生成业务数据的场景
2. **有更简单方法？** ✅ 当前方案已是最简：一个新节点类型 + 一个新字段 + 几处小改动
3. **会破坏什么？** ✅ 无破坏。无OpenPage节点的工作流→workflowOutputData为null→完成事件不携带→前端无影响
4. **项目真需要？** ✅ 必要。AI工作流的核心体验问题
5. **过度设计？** ✅ 没有。方案范围明确：仅后端，不做前端实施
6. **话题模糊？** ✅ 清晰。有完整的数据流设计和代码级别的实施方案
7. **学习注意事项？** ✅ 已学习。复用LLMAnswerNode模式；无数据库增删改操作需要bean方式；使用Lombok自动getter/setter

## 7. 向后兼容分析

- 无OpenPage节点的工作流 → `wfState.workflowOutputData`为null → 完成事件不携带 → 前端无影响
- 非JSON输入 → OpenPage节点正常调LLM生成友好文字，workflowOutputData存原始文本，前端解析失败静默忽略
- 新增组件不影响现有15种节点的执行逻辑
- `ChatResponseVo.workflowOutputData`字段已存在（第110行），无序列化兼容问题
- `convertWorkflowEventToChatResponse`中未知type走default分支返回空块，不会报错

## 8. 风险分析

| 风险 | 等级 | 缓解措施 |
|------|------|---------|
| WorkflowEngine.doOnComplete中emit失败 | 低 | tryEmitNext有返回值，可在日志中记录 |
| LLM生成的友好文字过长或不自然 | 低 | prompt限制100字以内，可后续调优 |
| 并发场景下wfState字段竞争 | 无 | 每个工作流实例独立的wfState，不存在共享 |
