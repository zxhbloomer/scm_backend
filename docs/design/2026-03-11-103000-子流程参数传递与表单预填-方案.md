# 子流程参数传递与表单预填 方案设计

## 一、问题描述

用户在 AI Chat 输入"帮我新增一个采购项目，项目名称为测试项目001"，系统打开了新增页面，但表单字段（项目名称）没有自动填入。

## 二、完整调用链路

```
用户输入: "帮我新增一个采购项目，项目名称为测试项目001"
  ↓
主流程"采购-项目管理-新增"(workflow_id=34)
  ├─ 临时知识库检索 → var_input_rag
  ├─ 项目管理知识库检索 → var_setting_rag（含字段规范文档）
  ├─ 合并结果节点(id=211) → var_nuts（JSON含用户输入+字段规范）
  └─ 子流程节点(id=255) "打开前端页面"
       input_config: var_nuts 已传入
       node_config.input_mapping: [] ← 空，var_nuts 没有映射到子流程
  ↓
子流程"通用-打开页面-流程"(workflow_id=42)
  开始节点(id=243): 只有 var_user_input，没有接收 var_nuts 的变量
  ↓
  节点248"打开前端页面"(OpenPage节点)
    var_outer = 开始节点.var_user_input（只有用户原始输入）
    prompt = ""（空，不调用LLM）
    ↓ 直接从上游MCP输出解析 path 和 page_mode
    输出: {"route":"/po/project","page_mode":"new"}  ← 没有 form_data
  ↓
前端 AiPageRouter.navigateToPage()
  query._ai_form_data = undefined ← 空
  ↓
aiPageActionMixin.$handleAiPageAction()
  formData = {} ← 空
  $bus.$emit('ai-form-prefill', {}) ← 空对象，不触发填充
  ↓
表单为空
```

## 三、根因分析

**根因1（主要）**：主流程子流程节点255的 `input_mapping` 为空，`var_nuts`（含字段规范的合并JSON）没有传入子流程。

**根因2**：子流程开始节点没有接收 `var_nuts` 的输入变量定义。

**根因3**：子流程节点248的 `prompt` 为空，即使数据传进来也不会提取 `form_data`。

**根因4（前端）**：`SubWorkflowNodeProperty.vue` 的参数映射输入框，`handleAddMapping` 新建的对象不是响应式的，导致输入框无法输入。

**根因5（后端）**：`SubWorkflowNode.mapInputs()` 完全不读取 `input_mapping` 配置，即使前端配置了映射也不生效。

## 五、方案设计

### 5.1 后端：SubWorkflowNode.mapInputs() 加第三步

```java
// 第三步：按 input_mapping 配置覆盖/补充参数
List<SubWorkflowNodeConfig.InputMapping> mappings = config.getInputMapping();
if (mappings != null && !mappings.isEmpty()) {
    for (SubWorkflowNodeConfig.InputMapping mapping : mappings) {
        String sourceKey = mapping.getSourceKey();
        String targetKey = mapping.getTargetKey();
        if (sourceKey == null || sourceKey.isEmpty() || targetKey == null || targetKey.isEmpty()) {
            continue;
        }
        for (NodeIOData input : state.getInputs()) {
            if (sourceKey.equals(input.getName())) {
                subInputs.put(targetKey, input.getContent().getValue());
                log.debug("input_mapping 参数传递: {} → {}", sourceKey, targetKey);
                break;
            }
        }
    }
}
```

### 5.2 前端：修复 handleAddMapping 响应式

```js
handleAddMapping () {
  const idx = this.nodeConfig.input_mapping.length
  this.nodeConfig.input_mapping.push({ source_key: '', target_key: '' })
  this.$set(this.nodeConfig.input_mapping, idx, this.nodeConfig.input_mapping[idx])
}
```

### 5.5 数据库：节点248 node_config 加 prompt

```
从以下数据中提取用户明确提到的表单字段值。

数据：${var_nuts}

规则：
1. 从数据的 input 字段读取用户说了什么
2. 从数据的 var_setting_rag 字段规范文档中找到对应的字段名
```

## 六、KISS原则7问题

1. **真问题**：是，日志已证实 form_data 为空
2. **更简单的方法**：已是最简，改动最小
3. **会破坏什么**：不会，input_mapping 为空时跳过，向后兼容
4. **真的需要**：是，核心功能缺失
5. **过度设计**：否，改动精准
6. **话题模糊**：否，根因明确
7. **学习了注意事项**：是，无新建实体/VO，无QueryWrapper

## 七、风险分析

| 风险 | 概率 | 缓解 |
|------|------|------|
| 子流程其他节点收到多余变量 | 低 | 子流程节点只引用自己 input_config 里声明的变量 |
| LLM prompt 提取字段名错误 | 中 | 字段规范文档已在 var_setting_rag 里，LLM有足够上下文 |
| 数据库更新后工作流缓存未刷新 | 低 | 重新保存工作流即可刷新 |
