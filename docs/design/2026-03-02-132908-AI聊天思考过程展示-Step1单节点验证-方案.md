# AI聊天思考过程展示 - Step1 单节点全链路验证方案

**文档日期**: 2026-03-02 13:29:08
**实施范围**: Step 1 - KnowledgeRetrieval节点全链路验证（知识库检索节点）

---

## 1. 需求背景

用户在SCM智能助手对话时，工作流执行过程中前端只显示"SCM正在思考..."，无法了解当前执行到哪个节点、执行结果如何。

Step 1 目标：打通完整链路，仅针对 **KnowledgeRetrieval（知识库检索）** 节点，实现：
- 执行时显示"正在检索知识库..."
- 执行完成后显示"检索知识库完成（命中X条）"

其他节点本步骤静默跳过，不展示。

---

## 2. 完整调用链路

```
用户发送消息
    ↓
ChatRouterController.streamChat() → SSE流
    ↓
WorkflowEngine.start()
    ↓
WorkflowEngine.executeWorkflow(false)
    ├── mainStateGraph.compile(compileConfig)  ← 注入NodeEventListener
    ├── app.graphResponseStream()              ← 原图执行流
    │       ↓ 每个节点执行时
    │   NodeEventListener.before(nodeUuid)    → sinkRef.get().tryEmitNext(node_start事件)
    │   [节点实际执行]
    │   NodeEventListener.after(nodeUuid)     → sinkRef.get().tryEmitNext(node_complete事件)
    │       ↓ 图完成时
    │   graphStream.doOnComplete()            → sinkRef.get().tryEmitComplete()
    └── Flux.merge(graphStream, sinkRef.asFlux())  ← 合并两个流
    ↓
WorkflowEventAdapter.toResponse(WorkflowEventVo)
    └── "node_start"/"node_complete" type → 填充 ChatResponseVo.nodeEventType 等字段
    ↓
SSE事件推送到前端
    ↓
aiChatService.js sendMessageStream() onNodeEvent(chatResponse)
    ↓
Vuex chat.js store → SET_WORKFLOW_PROCESS_NODE mutation
    ↓
MessageList.vue 渲染 ThinkingSteps.vue 组件
```

---

## 3. 方案设计

### 3.1 新增 SSE 事件类型

在现有 4 种事件类型（runtime, chunk, output, interrupt）基础上新增：

| 类型 | 触发时机 | 核心字段 |
|------|----------|----------|
| `node_start` | 节点开始执行前 | type, node(uuid), nodeName, nodeTitle, timestamp |
| `node_complete` | 节点执行完成后 | type, node(uuid), nodeName, nodeTitle, duration(ms), summary |

Step 1 中 `summary` 仅对 KnowledgeRetrieval 包含 `{"matchCount": N}`。

### 3.2 VISIBLE_NODES 过滤策略

Step 1 只处理 KnowledgeRetrieval 节点。NodeEventListener 内部判断：
```java
private static final Set<String> VISIBLE_NODES = Set.of("KnowledgeRetrieval");
// before()/after() 回调中：if (!VISIBLE_NODES.contains(componentName)) return;
```

通过 `wfNodes` + `components` 列表查找 componentName（`AiWorkflowComponentEntity.name`）。

#### 文件4：`WorkflowEngine.java`
路径：`scm-ai/.../workflow/WorkflowEngine.java`

**操作A**: 类字段新增 2 个

```java
// AtomicReference确保resume()调用时创建新Sink，订阅不冲突
private final AtomicReference<Sinks.Many<WorkflowEventVo>> sinkRef = new AtomicReference<>();
// 记录各节点开始时间，用于计算duration
private final ConcurrentHashMap<String, Long> nodeStartTimes = new ConcurrentHashMap<>();
```

**操作B**: 新增内部类 `NodeEventListener`

```java
private class NodeEventListener implements GraphLifecycleListener {

    private static final Set<String> VISIBLE_NODES = Set.of("KnowledgeRetrieval");

    @Override
    public void before(String nodeId, OverAllState state, RunnableConfig config, long curTime) {
        String componentName = findComponentName(nodeId);
        if (!VISIBLE_NODES.contains(componentName)) return;

        nodeStartTimes.put(nodeId, curTime);
        String nodeTitle = findNodeTitle(nodeId);
        Sinks.Many<WorkflowEventVo> sink = sinkRef.get();
        if (sink != null) {
            sink.tryEmitNext(WorkflowEventVo.createNodeStartData(nodeId, componentName, nodeTitle, curTime));
        }
    }

    @Override
    public void after(String nodeId, OverAllState state, RunnableConfig config, long curTime) {
        String componentName = findComponentName(nodeId);
        if (!VISIBLE_NODES.contains(componentName)) return;

        Long startTime = nodeStartTimes.remove(nodeId);
        long duration = (startTime != null) ? (curTime - startTime) : 0L;
        String nodeTitle = findNodeTitle(nodeId);
        Map<String, Object> summary = buildSummary(componentName, nodeId, state);

        Sinks.Many<WorkflowEventVo> sink = sinkRef.get();
        if (sink != null) {
            sink.tryEmitNext(WorkflowEventVo.createNodeCompleteData(nodeId, componentName, nodeTitle, duration, summary));
        }
    }

    private String findComponentName(String nodeId) {
        return wfNodes.stream()
            .filter(n -> nodeId.equals(n.getUuid()))
            .findFirst()
            .map(n -> components.stream()
                .filter(c -> c.getId().equals(n.getWorkflowComponentId()))
                .findFirst()
                .map(AiWorkflowComponentEntity::getName)
                .orElse(""))
            .orElse("");
    }

    private String findNodeTitle(String nodeId) {
        return wfNodes.stream()
            .filter(n -> nodeId.equals(n.getUuid()))
            .findFirst()
            .map(AiWorkflowNodeVo::getTitle)
            .orElse("");
    }

    private Map<String, Object> buildSummary(String componentName, String nodeId, OverAllState state) {
        if (!"KnowledgeRetrieval".equals(componentName)) return null;
        // 从state中读取节点输出的matchCount
        String outputKey = NODE_OUTPUT_KEY_PREFIX + nodeId;
        Object outputObj = state.value(outputKey).orElse(null);
        if (outputObj == null) return null;
        Map<String, Object> summary = new HashMap<>();
        if (outputObj instanceof Map) {
            Map<?, ?> outputMap = (Map<?, ?>) outputObj;
            Object matchCount = outputMap.get("matchCount");
            if (matchCount != null) {
                summary.put("matchCount", matchCount);
            }
        }
        return summary.isEmpty() ? null : summary;
    }
}
```

**操作C**: `compile()` 行改为注入 listener

```java
// 原来（第248行）:
app = mainStateGraph.compile();

// 改为:
Sinks.Many<WorkflowEventVo> newSink = Sinks.many().unicast().onBackpressureBuffer();
sinkRef.set(newSink);
CompileConfig compileConfig = CompileConfig.builder()
    .withLifecycleListener(new NodeEventListener())
    .build();
app = mainStateGraph.compile(compileConfig);
```

> 注意：CompileConfig无saverConfig，默认使用MemorySaver，与原`compile()`行为一致。

**操作D**: `executeWorkflow()` 改为 Flux.merge 合并两流

```java
// 原来（第304-325行）:
return graphStream
    .doOnComplete(() -> { ... updateWorkflowComplete(); })
    ...
    .flatMap(graphResponse -> handleGraphResponse(graphResponse)...)
    ...;

// 改为:
Sinks.Many<WorkflowEventVo> localSink = sinkRef.get();
return Flux.merge(
    graphStream
        .doOnComplete(() -> {
            log.info("[graphStream] 完成");
            updateWorkflowComplete();
            if (localSink != null) localSink.tryEmitComplete();
        })
        .doOnError(e -> {
            log.error("[graphStream] 错误", e);
            if (localSink != null) localSink.tryEmitError(e);
        })
        .doOnCancel(() -> log.warn("[graphStream] 被取消"))
        .flatMap(graphResponse -> handleGraphResponse(graphResponse)),
    localSink != null ? localSink.asFlux() : Flux.empty()
);
```

```js
SET_WORKFLOW_PROCESS_NODE(state, { messageId, nodeEvent }) {
    if (!state.workflowProcessNodes[messageId]) {
        state.workflowProcessNodes[messageId] = { steps: [] }
    }
    const steps = state.workflowProcessNodes[messageId].steps
    if (nodeEvent.nodeEventType === 'node_start') {
        steps.push({
            nodeUuid: nodeEvent.nodeUuid,
            nodeName: nodeEvent.nodeName,
            nodeTitle: nodeEvent.nodeTitle,
            status: 'running',
            timestamp: nodeEvent.nodeTimestamp,
            duration: null,
            summary: null
        })
    } else if (nodeEvent.nodeEventType === 'node_complete') {
        const step = steps.find(s => s.nodeUuid === nodeEvent.nodeUuid)
        if (step) {
            step.status = 'done'
            step.duration = nodeEvent.nodeDuration
            step.summary = nodeEvent.nodeSummary || null
        }
    }
}
```

```js
    }
}
```

## 5. 关于 KnowledgeRetrieval matchCount 数据来源

需确认 KnowledgeRetrieval 节点的输出数据结构中是否包含 `matchCount`。

查看路径：`scm-ai/.../workflow/node/KnowledgeRetrievalNode.java`

若当前输出 Map 中没有 `matchCount` 字段，则 `buildSummary()` 返回 null，前端展示"检索知识库完成"（无数量），功能仍正常。

**风险**: 如果 OverAllState.value(outputKey) API 不能在 `after()` 回调中可靠获取到刚执行完的节点输出，则 summary 会是 null（降级展示，不影响核心功能）。

## 6. KISS原则7问答

1. **这是个真问题还是臆想出来的？**
   真问题。用户感知"SCM正在思考..."时完全不知道进度，工作流可能执行30秒以上。

2. **有更简单的方法吗？**
   是。用 GraphLifecycleListener 拦截节点事件，通过已有 SSE 通道推送，不需要新接口/新表/新配置。

3. **会破坏什么吗？**
   不会。新增字段对前端可选消费；VISIBLE_NODES 过滤保证非KnowledgeRetrieval节点静默；现有流程未修改逻辑。

4. **当前项目真的需要这个功能吗？**
   是。Step 1 完成后验证链路可行再扩展，避免一次性大改动风险。

5. **这个问题过度设计了吗？**
   否。最简实现：1个listener内部类 + 2个factory方法 + 6个VO字段 + 1个小Vue组件。

6. **话题是否模糊，是否会导致幻觉？**
   否。GraphLifecycleListener API已读取源码确认，`compile(CompileConfig)`方法已确认存在。

7. **是否已学习代码实施注意事项？**
   是。不使用QueryWrapper，不使用Map代替VO，AtomicReference解决unicast subscriber限制。
