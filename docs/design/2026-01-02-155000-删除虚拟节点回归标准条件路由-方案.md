# Switcher条件路由多目标问题解决方案

## 一、问题诊断

### 1.1 问题表现
工作流ID=34中，Switcher节点"权限判断条件"配置了条件`pZkTvFMANeJ6mCvJ66dsACLrlgGxqgod`对应2个目标节点：
- 临时知识库（QWV9vpC10Lt7dL5Ilh2sgVDTwls2IJFp）
- 项目管理知识库检索（AbBFxq_lUJYT7OZohGI1UFzYN2mIkGbq）

但实际执行时，只有第一个节点被执行，第二个节点不执行。

### 1.2 根因分析

**Spring AI Alibaba框架限制**：
```java
// addConditionalEdges的路由函数只能返回单个字符串
stateGraph.addConditionalEdges(sourceNode,
    edge_async(state -> "single_target"),  // 只能返回一个目标！
    Map.of("single_target", "targetNode")
);
```

**并行分支的框架要求**：
根据官方文档，并行分支必须满足：
1. 多个分支从同一节点分叉
2. 所有分支**必须汇聚到同一个节点**

当前工作流结构：
```
权限判断条件 → 临时知识库 → 临时知识库检索（不同下游）
               ↓
               项目管理知识库检索 → 大模型（不同下游）
```

两个分支的下游节点不同，不符合框架的汇聚要求。

### 1.3 虚拟节点方案失败原因

**尝试的方案**：
```
Switcher ─条件路由→ 虚拟并行分发节点 ─普通边→ 临时知识库
                                    └─普通边→ 项目管理知识库检索
```

**失败原因**：
虚拟节点有2个出边，被框架识别为并行节点，但两个目标的下游不汇聚，`StateGraph.compile()`时报错：
```
parallel node [virtual_parallel_...] must have only one target
```

## 三、方案设计

### 3.1 方案选择

**删除虚拟节点逻辑，回归标准条件路由**

核心原则：
1. **接受框架限制**：一个条件只能路由到一个目标节点
2. **删除过度设计**：移除虚拟并行分发节点相关代码
3. **保持简洁性**：使用框架标准API `addConditionalEdges`

### 3.2 修改文件清单

**后端文件**：
1. `WorkflowEngine.java` - 删除虚拟节点逻辑
2. `WorkflowConstants.java` - 删除虚拟节点常量
3. `ParallelDispatchNode.java` - **删除整个文件**
4. `WfNodeFactory.java` - 删除ParallelDispatchNode注册

## 五、预期效果

**修改前**：
- Switcher条件匹配后，尝试创建虚拟节点路由到多个目标
- compile()时报错，工作流无法执行

- Switcher条件匹配后，路由到**第一个**配置的目标节点

- compile()成功，工作流正常执行

## 六、风险分析

### 6.1 向后兼容性

✅ **完全兼容**：
- 单目标条件分支：行为不变
- 多目标条件分支：执行第一个目标（原来也只执行虚拟节点，现在至少能正常工作）

### 6.2 用户影响

⚠️ **功能限制**：
- 用户如果期望一个条件触发多个节点，需要调整工作流设计
- 建议方案：使用显式并行节点或多个Switcher条件

### 6.3 代码质量

✅ **提升**：
- 删除约150行虚拟节点代码
- 消除不必要的复杂度
- 符合框架设计哲学

## 七、替代方案（用户侧）

如果用户确实需要一个条件触发多个分支，可以：

### 方案A：显式并行节点

```
权限判断条件 → 触发检索 → [并行分发节点] → 临时知识库
                                        → 项目管理知识库检索
```

### 方案B：多个条件

```
权限判断条件 → 条件A（临时知识库） → 临时知识库
              ↓
              条件B（项目管理知识库）→ 项目管理知识库检索
```

## 八、后续建议

3. **最佳实践**：推荐使用显式并行节点而非条件边多目标
