# 多End节点支持 — 图线性化方案

## 需求描述

工作流支持多个End节点（发散并行拓扑）。用户期望：

```
合并结果 → 美化返回 → 结束1（在聊天窗口显示友好文字）
合并结果 → 打开前端页面 → 结束2（触发业务弹窗）
```

两条路径并行发出，各自终止于不同的End节点。

## 问题根因

Alibaba框架 `CompiledGraph.compile()` 在验证并行边时（第210-221行）：

```java
// 检查并行分支的直接后继节点集合
var parallelNodeTargets = parallelNodeEdges.stream()
    .map(ee -> ee.target().id())
    .collect(Collectors.toSet());

if (parallelNodeTargets.size() > 1) {
    // 并行分支收敛到不同目标 → 抛出异常
    throw Errors.illegalMultipleTargetsOnParallelNode.exception(...);
}
```

验证逻辑：
- 合并结果 → [美化返回, 打开前端页面]（并行）
- 美化返回的直接后继：结束1
- 打开前端页面的直接后继：结束2
- parallelNodeTargets = {结束1, 结束2}，size=2 > 1 → 抛出异常

## 调用链路

```
用户设计工作流（含2个End节点）
      ↓
WorkflowEngine.init() → buildStateGraph()
      ↓
buildStateGraph() → 添加所有节点、边到StateGraph
      ↓
stateGraph.compile() → CompiledGraph.compile()
      ↓
CompiledGraph验证：并行分支后继必须相同
      ↓ (当前：报错)
illegalMultipleTargetsOnParallelNode
```

修复后：

```
buildStateGraph() → linearizeDivergentPaths()
      ↓
检测到 合并结果→[美化返回, 打开前端页面] 为发散并行
      ↓
修改内存中的 wfEdges：
  - 移除：合并结果→打开前端页面
  - 新增：结束1→打开前端页面
      ↓
buildStateGraph() 使用修改后的边构建：
  合并结果→美化返回→结束1→打开前端页面→结束2→__END__
      ↓
stateGraph.compile() → 通过（顺序链无汇聚问题）
      ↓
执行顺序：合并结果 → 美化返回 → 结束1 → 打开前端页面 → 结束2
```

## 方案设计

### 核心思路：图线性化（Graph Linearization）

在 `buildStateGraph()` 调用 `compile()` 之前，将发散并行路径转换为顺序链。

**原始边**（从数据库加载）：
```
合并结果 → 美化返回
合并结果 → 打开前端页面   ← 这条边会被移除
美化返回 → 结束1
打开前端页面 → 结束2
```

**线性化后边**（修改内存中的 wfEdges）：
```
合并结果 → 美化返回
美化返回 → 结束1
结束1 → 打开前端页面     ← 新增的链式边
打开前端页面 → 结束2
结束2 → __END__          ← 只有最后一个End节点连接__END__
```

注意：结束1 不连接 __END__（标记为中间End节点）。

### 代码变更详情

#### 变更1：buildStateGraph() 修改

在现有第3步（计算conditionalNodeUuids）之后，第4步（添加边）之前，插入第3.5步：

```java
// 3.5 线性化发散并行路径，绕过Alibaba框架的汇聚验证
Set<String> intermediateEndNodes = linearizeDivergentPaths(conditionalNodeUuids);
```

在第6步（添加End→__END__边）中，增加过滤逻辑：

```java
// 6. 找到所有结束节点，添加到END的边（跳过中间End节点）
for (AiWorkflowNodeVo node : wfNodes) {
    if (isEndNode(node) && !intermediateEndNodes.contains(node.getUuid())) {
        addEdgeToStateGraph(stateGraph, node.getUuid(), END);
    }
}
```

#### 变更2：linearizeDivergentPaths() 新增方法

```
职责：检测发散并行分叉，将其转为顺序链
输入：conditionalNodeUuids（需要排除的条件分支节点）
输出：中间End节点UUID集合（这些End节点不应连接__END__）
副作用：修改 wfEdges 列表（内存中，不影响数据库）
```

算法步骤：
1. 按source分组所有非条件边
2. 找出有2+个出边的分叉节点
3. 对每个分叉，追踪每条路径到其End节点
4. 检查各路径是否终止于不同End节点（发散）
5. 若发散：移除fork到后续路径起点的直接边，改为前一路径End→后续路径起点的链式边
6. 标记中间End节点

#### 变更3：tracePathToEnd() 新增方法

```
职责：从指定节点开始，沿第一条出边追踪直到End节点或无出边
输入：startNodeUuid
输出：路径上的节点UUID列表（含起点和End节点）
```

## KISS原则7问题回答

1. **这是个真问题还是臆想出来的？**
   真问题。用户运行含2个End节点的工作流时，Alibaba框架在compile阶段直接报错，工作流无法执行。

2. **有更简单的方法吗？**
   当前方案是最简方案。只改一个文件，两个私有方法，约70行代码。不需要修改Alibaba框架、不需要修改End节点逻辑、不需要新建文件。

3. **会破坏什么吗？**
   不会。线性化只在检测到发散并行时才触发（endNodeUuids.size() > 1）。单End节点、汇聚并行、条件分支均不受影响，有明确的跳过条件。

4. **当前项目真的需要这个功能吗？**
   是。用户明确提出需要两个End节点分别输出友好文字和触发弹窗，这是OpenPage节点完整方案的必要组成部分。

5. **这个问题过度设计了吗？**
   没有。方案是最小改动实现最大价值：一个文件，两个方法，解决框架层面的根本限制。

6. **话题是否模糊，是否会导致幻觉？**
   方案基于实际源码验证（CompiledGraph.java:210-221、WorkflowEngine.java全文、EndNode.java），非推测。

7. **是否学习了代码实施的注意事项？**
   是。wfEdges修改仅在内存中，不写数据库（符合规范）。无QueryWrapper（符合第24条）。无新文件创建（符合最小改动原则）。

## 向后兼容验证

| 场景 | 处理路径 | 结果 |
|------|----------|------|
| 单End节点工作流 | edgesBySource无分叉节点 | 返回空集，零改动 |
| 汇聚并行（两路同一End） | endNodeUuids.size()=1 → 跳过 | 返回空集，框架原生处理 |
| 条件分支（Switcher/Classifier） | conditionalNodeUuids过滤 | 不参与线性化 |
| 2个End节点发散并行 | 线性化为顺序链 | 正常执行 |
| 3个End节点发散并行 | 递归链式：End1→Path2→End2→Path3→End3 | 正常执行 |

## 执行流说明

线性化后执行顺序（以用户案例为例）：

1. `合并结果` 执行 → 输出存入 `node_output_<T_uuid>` → SSE事件发送
2. `美化返回` 执行 → 引用合并结果输出 → SSE事件发送
3. `结束1` 执行 → 模板渲染 → SSE事件发送 → **前端显示友好文字** → 不终止图
4. `打开前端页面` 执行 → 引用合并结果输出（通过ref_inputs） → 设置ai_open_dialog_para → SSE事件发送
5. `结束2` 执行 → 模板渲染 → SSE事件发送 → **图终止**
6. `doOnComplete` → 发送 `ai_open_dialog_para` 事件 → 关闭Sink → **前端触发弹窗**

数据隔离保障：每个节点通过 `ref_inputs` 引用特定上游节点UUID的数据，与执行顺序无关。打开前端页面节点通过 `ref_inputs` 引用合并结果节点（不是结束1节点），数据完全正确。

## 风险分析

| 风险 | 概率 | 影响 | 缓解 |
|------|------|------|------|
| wfEdges修改影响processConditionalEdges | 低 | 中 | conditionalNodeUuids过滤确保条件分支边不被处理 |
| tracePathToEnd死循环 | 极低 | 中 | visited集合防止循环路径 |
| 中间End节点的SSE事件被前端误处理 | 低 | 低 | 前端按componentName="End"处理，多个End节点输出均可正常显示 |
| 线性化后打开前端页面无法访问合并结果数据 | 无 | 无 | ref_inputs通过UUID精确引用，与执行顺序无关 |
