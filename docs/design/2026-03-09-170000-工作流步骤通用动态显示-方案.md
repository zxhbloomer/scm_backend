# 工作流执行步骤通用动态显示 - 设计方案

## 1. 需求概述

用户在AI聊天中发送消息后，LLM路由动态选择某个工作流（agent）执行。前端ThinkingSteps组件需要通用地展示该工作流从Start到End的完整执行过程。

### 显示效果（通用，适用于任何被选中的工作流）

```
问题分析完成
|
调用agent：[工作流标题]
  ●开始：var_user_input 用户输入
  ●[节点标题]：[执行结果]
  ●[节点标题]：[执行结果]
  ●[结束节点标题]
调用agent完成
```

### 关键约束

1. 完全通用：不针对任何特定工作流定制，所有节点标题、参数信息都从后端动态获取
2. Start/End节点不需要show_process_output，它们是纯结构性节点
3. 三种场景一致：执行中（SSE流式）、执行完成、刷新加载历史消息

## 2. 完整调用链路

```
用户发送消息
  → AiConversationController.chatStream()
    → WorkflowRoutingService.routeAndExecute()
      → LLM路由选择工作流
      → WorkflowEngine.run()
        → 发送 runtime 事件（含workflowTitle）
        → NodeEventListener.before() → 发送 node_start 事件
        → 节点执行
        → NodeEventListener.after() → 发送 node_complete 事件（含summary）
        → ... 循环直到End节点
  → WorkflowEventAdapter.convert() → ChatResponseVo
    → SSE流推送到前端
      → aiChatService.js 解析SSE
        → onNodeEvent回调
          → chat.js SET_WORKFLOW_PROCESS_NODE mutation
            → ThinkingSteps.vue 渲染
```

### 4.2 WorkflowEngine.java - buildSummary 加入 Start case

在 buildSummary 方法的 switch 中新增 Start case，携带Start节点的输入参数定义列表。

```java
case "Start": {
    // 携带Start节点的输入参数定义（变量名+标题），前端动态显示
    AiWorkflowNodeVo node = wfNodes.stream()
        .filter(n -> nodeId.equals(n.getUuid())).findFirst().orElse(null);
    if (node != null && node.getInputConfig() != null
            && node.getInputConfig().getUserInputs() != null) {
        summary = new HashMap<>();
        List<Map<String, String>> params = new ArrayList<>();
        for (AiWfNodeIOVo io : node.getInputConfig().getUserInputs()) {
            Map<String, String> p = new HashMap<>();
            p.put("name", io.getName());
            p.put("title", io.getTitle());
            params.add(p);
        }
        summary.put("params", params);
    }
    break;
}
```

End节点不需要特殊summary，不加case（走default返回null）。

```

```

## 5. 前端修改详细设计

### 5.1 chat.js - runtime事件处理

**当前代码**（第383-392行）：
```js
onNodeEvent: (nodeEvent) => {
  if (nodeEvent.nodeEventType === 'runtime') {
    return  // 当前直接忽略
  }
  commit('SET_WORKFLOW_PROCESS_NODE', { messageId: aiMessageId, nodeEvent })

```

**修改后**：
```js
onNodeEvent: (nodeEvent) => {
  if (nodeEvent.nodeEventType === 'runtime') {
    // runtime事件：插入"调用agent：xxx"包裹行
    commit('SET_WORKFLOW_PROCESS_NODE', {
      messageId: aiMessageId,
      nodeEvent: {
        nodeEventType: 'node_start',
        nodeUuid: '__agent_call__',
        nodeName: 'AgentCall',
        nodeTitle: nodeEvent.workflowTitle || '工作流',
        nodeTimestamp: Date.now()
      }
    })
    return
  }
  commit('SET_WORKFLOW_PROCESS_NODE', { messageId: aiMessageId, nodeEvent })
},
```

### 5.2 chat.js - FLUSH_PENDING_NODE_COMPLETE

在流结束时，除了刷新最后一个节点的pendingComplete，还需要把agent包裹行标记为done。

**在现有FLUSH逻辑末尾追加**：
```js
// 标记agent包裹行完成
const agentStep = processData.steps.find(s => s.nodeUuid === '__agent_call__')
if (agentStep && agentStep.status !== 'done') {
  agentStep.status = 'done'
  // duration = 所有内部步骤的duration之和
  const innerDuration = processData.steps
    .filter(s => s.nodeUuid !== '__agent_call__' && s.nodeUuid !== '__virtual_analysis__')
    .reduce((sum, s) => sum + (s.duration || 0), 0)
  agentStep.duration = innerDuration
}
```

### 5.3 ThinkingSteps.vue - getStepText 方法修改

在现有getStepText方法中，加入Start/End/AgentCall的处理逻辑。所有显示文本来自后端动态数据。

```js
getStepText (step) {
  const name = step.nodeName

  // 虚拟问题分析步骤
  if (step.nodeUuid === '__virtual_analysis__') {
    return step.status === 'running' ? '问题分析中...' : '问题分析完成'
  }

  // Agent包裹行
  if (step.nodeUuid === '__agent_call__') {
    return step.status === 'running'
      ? `调用agent：${step.nodeTitle}...`
      : `调用agent：${step.nodeTitle} 完成`
  }

  // Start节点：显示标题 + 参数列表（从后端summary.params动态获取）
  if (name === 'Start') {
    if (step.status === 'running') return `${step.nodeTitle || '开始'}...`
    const params = step.summary && step.summary.params
    if (params && params.length > 0) {
      const paramText = params.map(p => `${p.name} ${p.title}`).join(', ')
      return `${step.nodeTitle || '开始'}：${paramText}`
    }
    return step.nodeTitle || '开始'
  }

  // End节点：直接用后端传来的nodeTitle
  if (name === 'End') {
    return step.nodeTitle || '结束'
  }

  // Classifier → 用节点标题显示（保持不变）
  // Answer/LLM → 生成答案（保持不变）
  // 其他节点 → 标题 + 摘要（保持不变）
  // ... 现有逻辑不变 ...
}
```

### 5.4 ThinkingSteps.vue - isCompleted 计算属性修改

需要排除agent包裹行对完成判断的干扰。

**当前逻辑**：最后一步是Answer/LLM/OpenPage且done → 完成

**修改后**：最后一个非AgentCall步骤是Answer/LLM/OpenPage/End且done → 完成

```js
isCompleted () {
  if (!this.isAllDone) return false
  if (this.streamComplete) return true
  // 找最后一个非agent包裹行的步骤
  const realSteps = this.steps.filter(s => s.nodeUuid !== '__agent_call__')
  const lastStep = realSteps[realSteps.length - 1]
  return lastStep && (
    lastStep.nodeName === 'Answer' ||
    lastStep.nodeName === 'LLM' ||
    lastStep.nodeName === 'OpenPage' ||
    lastStep.nodeName === 'End'
  ) && lastStep.status === 'done'
},
```

### 5.5 ThinkingSteps.vue - agent包裹行样式

agent包裹行需要视觉上区分于普通步骤节点，作为一个"分组标题"存在。

**渲染逻辑**：在template的v-for中，判断step.nodeUuid === '__agent_call__'时使用不同样式：
- 圆点改为agent图标（或使用不同颜色的圆点）
- 文字加粗
- 内部步骤（Start/McpTool/Classifier/OpenPage/End）保持现有缩进

### 5.6 历史消息兼容

workflowSteps持久化到数据库时，agent包裹行和Start/End步骤都会被序列化保存。
loadMessages恢复时，JSON.parse后数据结构完全一致，ThinkingSteps直接渲染，无需额外处理。

## 7. KISS原则7问题回答

1. **这是个真问题还是臆想出来的？** → 真问题。用户需要看到工作流执行的完整过程，当前缺少Start/End节点和agent包裹行。
2. **有更简单的方法吗？** → 当前方案已是最简：后端加2个节点类型到VISIBLE_NODES + 1个buildSummary case，前端改3处逻辑。
3. **会破坏什么吗？** → 不会。Start/End事件是新增的，现有节点事件逻辑不变。前端agent包裹行是新增步骤类型，不影响现有步骤渲染。
4. **当前项目真的需要这个功能吗？** → 需要。这是AI聊天核心体验，用户需要了解工作流执行进度。
5. **过度设计了吗？** → 没有。方案改动量小，逻辑清晰。
6. **话题是否模糊？** → 不模糊。需求明确，日志数据已验证。
7. **是否学习了代码实施注意事项？** → 是。不使用QueryWrapper，SQL用别名；不硬编码前端配置；复用现有事件机制。

## 8. 风险分析

| 风险 | 影响 | 缓解措施 |
|------|------|----------|
| Start/End节点执行极快，node_start和node_complete几乎同时到达 | 前端可能看不到running状态 | 可接受，pendingComplete缓冲机制已处理 |
| 历史消息中旧数据没有agent包裹行和Start/End步骤 | 旧消息显示不变 | 向后兼容，旧数据无这些步骤，ThinkingSteps正常渲染 |
