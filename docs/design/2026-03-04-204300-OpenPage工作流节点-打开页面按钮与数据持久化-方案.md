# OpenPage工作流节点 - 打开页面按钮与数据持久化 设计方案

> 日期: 2026-03-04 20:43:00
> 版本: v1.0

## 一、需求概述

工作流包含两条并行分支：
- **上分支**：美化返回(LLM) → 结束2 → 聊天显示友好文字
- **下分支**：临时知识库检索 → OpenPage节点 → 结束1 → 存JSON到侧通道，自动打开业务弹窗

三个核心需求：
1. **自动打开页面**：OpenPage节点存在时，根据JSON中的`ai_new_route`自动打开业务弹窗
2. **美化返回**：聊天窗口显示LLM美化后的友好文字，不显示原始JSON
3. **打开页面按钮**：聊天消息底部增加"打开页面"按钮，支持重复打开，基于消息时间戳判断时效（24小时）

## 二、完整调用链路

### 2.1 实时执行链路（当前SSE流）

```
前端发送消息
    → AiConversationController.chatStream()
        → WorkflowRoutingService.executeWorkflowByUuid()
            → WorkflowStarter.streaming()
                → WorkflowEngine.run()
                    → StateGraph执行（两条分支并行）
                        → 上分支：美化返回(LLM Answer) → End2
                        → 下分支：OpenPage节点 → End1
                    → doOnComplete()
                        → 检查 wfState.getWorkflowOutputData()
                        → 通过 Sink 发送 workflow_output_data 事件
            ← 返回 Flux<WorkflowEventVo>
        → eventFlux.map() 捕获事件
            → "workflow_output_data" → capturedWorkflowOutputData[0]
            → "chunk" → 转为 ChatResponseVo 内容块
        → Flux.concat(contentFlux, 完成事件)
            → 完成事件: isComplete=true, workflowOutputData=JSON
    ← 返回 Flux<ChatResponseVo> (SSE流)
        → AiConversationController.handleResponse()
            → isComplete=true 时保存消息:
                → saveContent(conversationId, 1, 用户消息, operatorId, null)
                → saveContent(conversationId, 2, AI美化回复, operatorId, runtimeUuid)
    ← 前端接收 SSE
        → onComplete(finalContent, chatResponse)
            → SET_PENDING_AI_DIALOG_OUTPUT → watch → AiBusinessDialogLoader.open() → 自动打开弹窗
            → UPDATE_MESSAGE → 消息附带 workflowOutputData
```

### 2.2 历史消息加载链路

```
前端进入会话
    → aiChatService.getAiChatDetail(conversationId)
        → GET /api/v1/ai/conversation/chat/list/{conversationId}
            → AiConversationService.chatList()
                → ExtAiConversationContentMapper.selectByConversationId()
                    → SELECT ... FROM ai_conversation_content WHERE conversation_id = ?
    ← 返回消息列表
        → chat.js loadMessages()
            → 格式化消息（需要包含 workflow_output_data）
        → MessageList.vue 渲染
            → 如果消息有 workflow_output_data 且在24小时内 → 显示"打开页面"按钮
```

### 2.3 当前缺失环节（本次需要补全的）

| 环节 | 当前状态 | 需要做的 |
|------|---------|---------|
| `ai_conversation_content`表 | 无`workflow_output_data`列 | ALTER TABLE 新增字段 |
| `AiConversationContentEntity` | 无`workflowOutputData`字段 | 新增字段 |
| `AiConversationContentService.saveContent()` | 不保存`workflowOutputData` | 新增参数，保存时写入 |
| `AiConversationController.handleResponse()` | 不传递`workflowOutputData` | 从response取值传给saveContent |
| `ExtAiConversationContentMapper` | SELECT不包含`workflow_output_data` | SQL新增该字段 |
| `AiConversationContentVo` | 无`workflow_output_data`字段 | 新增字段 |
| 前端 `chat.js` loadMessages | 不解析`workflow_output_data` | 新增字段映射 |
| 前端 `MessageList.vue` | 无"打开页面"按钮 | 新增按钮渲染逻辑 |
| `OpenPageNode.java` | 包含LLM调用逻辑 | 精简为纯数据存储 |

## 五、KISS原则7问题

1. **这是个真问题还是臆想出来的？**
   真问题。当前End节点把原始JSON输出到聊天窗口，用户体验极差。按钮和持久化是用户明确提出的需求。

2. **有更简单的方法吗？**
   已是最简方案。只新增1个数据库字段、修改1个saveContent方法签名、前端1个按钮组件。没有新增Service/Mapper/Controller。

3. **会破坏什么吗？**
   `saveContent()`签名变更影响2处调用点，都在同一个Controller文件内。ALTER TABLE新增列不影响现有数据。向后兼容。

4. **当前项目真的需要这个功能吗？**
   是的。OpenPage是AI辅助创建业务数据的核心节点。

5. **这个问题过度设计了吗？**
   不是。10个文件变更，大部分是小改（新增1个字段）。核心改动是saveContent新增参数和MessageList新增按钮。

6. **话题是否模糊，是否会导致幻觉的产生？**
   不会。所有调用链路已通过代码阅读和数据库查询验证。

7. **是否已经学习了关于代码实施的注意事项的内容？**
   是的。关键注意事项：
   - 第17条：插入使用bean操作（saveContent已符合）
   - 第19条：SQL使用AS别名方案（ExtMapper已有pattern）
   - 第21条：使用BeanUtils.copyProperties（saveContent已使用）
   - 第23条：使用UuidUtil.createShort()（saveContent已使用）
   - 第24条：查询使用sql在mapper中（ExtMapper已有pattern）
   - 第27条：sql使用"""+sql+"""方式（ExtMapper已有pattern）

## 六、风险分析

| 风险 | 等级 | 缓解措施 |
|------|------|---------|
| saveContent签名变更导致编译错误 | 低 | 只有2处调用，都在同一个Controller文件 |
| ALTER TABLE在大表上耗时 | 低 | ai_conversation_content表数据量不大 |
| 前端按钮时效判断时区问题 | 低 | 使用服务器返回的c_time，前端统一用new Date()比较 |
| 历史消息workflow_output_data为null | 无 | 前端已做null判断，按钮不显示 |
