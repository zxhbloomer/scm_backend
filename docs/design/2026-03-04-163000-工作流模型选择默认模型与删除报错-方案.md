# 工作流模型选择：默认模型选项与删除报错

## 问题背景

工作流中有**8个节点类型**使用了LLM模型选择。当用户在"AI模型设置"中删除或停用了被工作流引用的模型后：

**当前行为**：后端静默回退到系统默认模型，用户毫不知情
**期望行为**：
1. 所有使用模型的节点都支持显式选择"默认模型"，明确表达意图
2. 如果指定了具体模型，但模型被删/停用 → 启动工作流时报错
3. 如果选了"默认模型"，但系统默认模型被删 → 启动工作流时报错

## 影响范围深度分析

### 使用LLM模型的8个节点类型

| 节点 | 后端调用路径 | 前端选择组件 | 模型字段 |
|------|------------|-------------|---------|
| Answer（LLM回答） | `WorkflowUtil.streamingInvokeLLM()` → `resolveModelConfig()` | WfLLMSelector | model_name |
| Classifier（分类器） | `WorkflowUtil.invokeLLM()` → `resolveModelConfig()` | WfLLMSelector | model_name |
| McpTool（MCP工具） | `WorkflowUtil.streamingInvokeLLM()` → `resolveModelConfig()` | WfLLMSelector | model_name |
| OpenPage（打开前端页面） | `WorkflowUtil.streamingInvokeLLM()` → `resolveModelConfig()` | WfLLMSelector | model_name |
| KeywordExtractor（关键词提取） | `WorkflowUtil.streamingInvokeLLM()` → `resolveModelConfig()` | WfLLMSelector | model_name |
| FaqExtractor（FAQ提取） | `WorkflowUtil.streamingInvokeLLM()` → `resolveModelConfig()` | WfLLMSelector | model_name |
| TempKnowledgeBase（临时知识库） | `WorkflowUtil.streamingInvokeLLM()` → `resolveModelConfig()` | WfLLMSelector | model_name |
| KnowledgeRetrieval（知识库检索） | 独立路径（`graphRetrievalService`），已有空值报错 | WfLLMSelector | graph_model_name |

### 架构优势：2个共用入口

所有节点共用2个入口，**改动2个文件即可覆盖全部8个节点**：

```
前端 WfLLMSelector（共用组件）         后端 WorkflowUtil.resolveModelConfig()（共用方法）
┌──────────────────────┐              ┌──────────────────────────────┐
│ 所有8个节点的属性面板    │              │ 7个节点的LLM调用              │
│ 都使用 WfLLMSelector   │  ──────→    │ 都走 resolveModelConfig()    │
│                        │              │                              │
│ 改1处 = 8个节点获益     │              │ 改1处 = 7个节点获益           │
└──────────────────────┘              └──────────────────────────────┘
                                       KnowledgeRetrieval独立路径，
                                       已有空值报错，无需改动
```

## 数据流分析

```

┌─────────────────┐                                        ┌────────────────────────┐

│ 选择"gj-deepseek"│ ──→ model_name = "gj-deepseek" ──→  │ getModelConfigByName()  │
```

## 后端修改

### 1. WorkflowUtil.java — resolveModelConfig()

**路径**：`scm-ai/src/main/java/com/xinyirun/scm/ai/workflow/WorkflowUtil.java`

**修改前**（第448-460行）：
```java
private static AiModelConfigVo resolveModelConfig(String modelName, ...) {
    if (StringUtils.isNotBlank(modelName)) {
        try {
            return aiModelConfigService.getModelConfigByName(modelName);
        } catch (Exception e) {
            log.warn("指定模型 [{}] 未找到，使用系统默认模型", modelName);  // 静默回退
        }
    }
    return aiChatBaseService.getModule(request, null);
}
```

**修改后**：
```java
private static AiModelConfigVo resolveModelConfig(String modelName, ...) {
    if (StringUtils.isNotBlank(modelName)) {
        // 指定了具体模型：必须找到，否则报错
        return aiModelConfigService.getModelConfigByName(modelName);
    }
    // 未指定模型（"默认模型"选项）：使用系统默认
    return aiChatBaseService.getModule(request, null);
}
```

**变更说明**：
- 移除try-catch，让`getModelConfigByName()`的RuntimeException直接向上传播
- `getModelConfigByName()`已有清晰错误信息："模型配置不存在或未启用: xxx"
- 外层`streamingInvokeLLM()`的catch会包装为"LLM 流式调用失败: 模型配置不存在或未启用: xxx"
- 空modelName路径不变，`getModule()`内部找不到默认模型也会抛异常："未配置默认LLM模型"
- **影响节点**：Answer、Classifier、McpTool、OpenPage、KeywordExtractor、FaqExtractor、TempKnowledgeBase（共7个）
- **不影响**：KnowledgeRetrieval（独立路径，已有自己的验证逻辑）

## 前端修改

### 1. WfLLMSelector.vue — 增加"默认模型"选项

**路径**：`src/components/70_ai/components/workflow/components/WfLLMSelector.vue`

**修改内容**：
1. 在computed中新增`allOptions`，将"默认模型"插入到llmOptions头部
2. template中使用`allOptions`替代`llmOptions`
3. `handleSelect`中处理"默认模型"选择（value为-1），emit空字符串
4. `updateSelectedModelId`中处理空modelName（选中"默认模型"，value设为-1）

**"默认模型"选项定义**：
```javascript
{ label: '默认模型', value: -1, provider: '系统配置' }
```

当选择"默认模型"时，`model_name`存储为空字符串`''`。后端收到空字符串后走系统默认模型路径。

**影响节点**：所有8个使用WfLLMSelector的节点属性面板自动获得"默认模型"选项：
- AnswerNodeProperty.vue
- ClassifierNodeProperty.vue
- McpNodeProperty.vue
- OpenPageNodeProperty.vue
- KeywordExtractorNodeProperty.vue
- FaqExtractorNodeProperty.vue
- TempKnowledgeBaseNodeProperty.vue
- KnowledgeRetrievalNodeProperty.vue（图谱模型选择）

### 2. WfLLMSelector.vue — 已保存模型被删除时的前端提示

**问题场景**：工作流已保存`model_name = "gj-deepseek"`，但该模型被管理员删除/停用。用户打开工作流编辑器时：

**当前行为**（有缺陷）：
```javascript
// updateSelectedModelId("gj-deepseek")
const model = this.llmOptions.find(opt => opt.label === "gj-deepseek")
// 找不到 → model = undefined
this.selectedModelId = null  // 下拉框显示空白！用户不知道原来选的是什么
```

**期望行为**：
- 下拉框下方显示警告提示："模型 gj-deepseek 已不可用，请重新选择"
- 用户可以选择"默认模型"或其他可用模型

**实现方式**：
1. 新增`data`字段`unavailableModelName`，记录不可用的模型名称
2. `updateSelectedModelId`中，当modelName有值但在列表中找不到时，设置`unavailableModelName`
3. template中，当`unavailableModelName`有值时，在下拉框下方显示`el-alert`警告

```javascript
// updateSelectedModelId 中新增逻辑
if (modelName && !model) {
    this.unavailableModelName = modelName  // 记录不可用的模型名
} else {
    this.unavailableModelName = ''
}
```

```html
<!-- template中在el-select下方新增 -->
<el-alert
  v-if="unavailableModelName"
  :title="'模型 ' + unavailableModelName + ' 已不可用，请重新选择'"
  type="warning"
  :closable="false"
  show-icon
  style="margin-top: 8px;"
/>
```

## KISS原则7问题

1. **真问题还是臆想？** ✅ 真问题 — 模型被删后静默切换，用户不知情，影响全部8个节点类型
2. **有更简单的方法吗？** ✅ 已是最简 — 利用共用入口，改2个文件覆盖全部8个节点
3. **会破坏什么吗？** ✅ 无破坏 — 现有工作流如果model_name为空，行为不变（用默认）；如果model_name有值且模型存在，行为不变。唯一变化：指定模型被删后从"静默回退"变为"报错"
4. **真的需要吗？** ✅ 需要 — 避免用户不知情时使用错误模型
5. **过度设计？** ✅ 不是 — 2个文件、总共改动约20行，覆盖8个节点
6. **话题模糊？** ✅ 不模糊 — 需求明确，实现路径清晰
7. **学习了代码实施注意事项？** ✅ 是 — 复用现有代码逻辑，无新Entity/VO/Mapper
