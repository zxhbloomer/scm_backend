# 工作流LLM流式输出改造 - 设计方案

## 1. 问题描述

当前SCM-AI工作流中LLM节点（LLMAnswerNode）调用`WorkflowUtil.streamingInvokeLLM()`时，
使用`blockLast()`阻塞等待LLM全部完成后一次性输出，导致前端看不到打字机效果（逐token流式输出）。

**根因定位**：`WorkflowUtil.java` 第256行 `.blockLast()` 阻塞了整个`Flux<ChatResponse>`，
token被累积到`fullResponse`后一次性存入`nodeState.outputs`。

## 2. 改造范围

**核心原则**：只有终端节点需要流式输出，中间节点保持blockLast。

| 分类 | 节点 | 当前行为 | 改造后 |
|------|------|---------|--------|
| 终端节点 | LLMAnswerNode | blockLast，一次性输出 | blockLast + 同时发送chunk事件（打字机） |
| 终端节点 | OpenPageNode | 非流式invokeLLM | 不改（静默调LLM生成JSON） |
| 中间节点 | ClassifierNode | 非流式invokeLLM | 不改 |
| 中间节点 | FaqExtractorNode | blockLast, silentMode=false | silentMode改为true |
| 中间节点 | KeywordExtractorNode | blockLast, silentMode=false | silentMode改为true |
| 中间节点 | McpToolNode | blockLast, silentMode可变 | 不改（已有silentMode控制） |

参考项目（Dify、FastGPT、Spring AI Alibaba）一致：中间节点阻塞获取完整结果用于路由决策，终端节点流式输出。

## 3. 技术方案：Sink直推

### 3.1 方案选型

分析了两种实现路径后选择Sink直推：

| 对比 | EmbedFlux路径 | Sink直推路径（选定） |
|------|-------------|---------------------|
| 数据流 | Flux穿透框架管道 | doOnNext直接发送Sink |
| 改动文件数 | 5个 | 5个（但改动量小） |
| 风险 | 高（需改handleGraphResponse核心逻辑） | 低（只增加chunk发送） |
| blockLast | 移除 | 保留（同时发送chunk） |
| 前端效果 | 打字机 | 打字机（一致） |

### 3.2 核心思路

保持`blockLast()`确保节点等待LLM完成（runNode需要完整outputs），
但在`doOnNext()`回调中通过Sink直接发送chunk事件给前端。

Sink通道已存在：`NodeEventListener`已通过`sinkRef.get()`发送`node_start`/`node_complete`事件。

### 3.3 数据时序

```
t0: LLMAnswerNode.onProcess() 调用 streamingInvokeLLM()
t1: LLM返回token -> doOnNext -> Sink.tryEmitNext(chunk) -> 前端显示"你"
t2: LLM返回token -> doOnNext -> Sink.tryEmitNext(chunk) -> 前端显示"你好"
...
tN: LLM完成 -> blockLast返回 -> fullResponse存入nodeState.outputs -> runNode返回
tN+1: 框架触发handleGraphResponse -> processNodeOutput -> DB写入
tN+2: 框架调度下一个节点(End)
```

## 4. 文件变更清单

### 4.1 WfState.java

新增两个字段：

```java
/**
 * 事件Sink引用，供LLM流式调用时发送chunk事件到前端
 */
private transient Sinks.Many<WorkflowEventVo> eventSink;

/**
 * 已通过chunk事件流式输出的节点UUID集合
 * 用于handleGraphResponse中跳过output事件，避免内容重复
 */
private final Set<String> streamedNodeUuids = ConcurrentHashMap.newKeySet();

public void markNodeStreamed(String nodeUuid) {
    streamedNodeUuids.add(nodeUuid);
}

public boolean hasNodeStreamed(String nodeUuid) {
    return streamedNodeUuids.contains(nodeUuid);
}
```

## 5. 前端无需改动

chunk事件从后端到前端的完整链路已存在，只是从未被触发：

1. WorkflowEventVo.createChunkData() - 已定义（第65行），从未被调用
2. WorkflowRoutingService.convertWorkflowEventToChatResponse() - 已有chunk类型处理（第1392行）
3. 前端aiChatService.js - 已有流式content累积逻辑
4. 前端chat.js store - 已有onContent追加渲染

## 6. 重复内容问题处理

**问题**：chunks逐token发送后，handleGraphResponse仍发送output事件。
convertWorkflowEventToChatResponse对Answer/LLM节点会提取完整文本，前端收到重复内容。

**解决**：WfState.streamedNodeUuids记录已流式节点，handleGraphResponse中跳过已流式节点的output事件。
DB写入不受影响（processNodeOutput在跳过判断之前执行）。

## 7. 向后兼容分析

| 场景 | 影响 |
|------|------|
| 无LLM节点的工作流 | 无影响，streamingInvokeLLM不被调用 |
| 中间节点(Classifier等) | 无影响，使用非流式invokeLLM或silentMode=true |
| OpenPageNode | 无影响，使用非流式invokeLLM |
| McpToolNode | 受益（silentMode=false时也有打字机） |
| 子工作流(SubWorkflow) | 无影响，eventSink为null |
| eventSink为null | 安全，tryEmitNext前已判null |

## 8. KISS原则评估

1. 真问题：用户明确反馈"一下子全部显示出来了，不像流式输出"
2. 最简方案：核心只有一行tryEmitNext，总改动约22行
3. 零破坏：保留blockLast，不改框架调度逻辑，不改前端
4. 必要性：流式输出是AI对话产品的基本体验要求
5. 信息充足：已深度分析framework源码（NodeExecutor/StreamingOutput/OutputType）
6. 话题明确：有具体代码定位和日志佐证
7. 已学习注意事项：复用现有createChunkData/convertWorkflowEventToChatResponse基础设施
