# Switcher条件分支多目标并行执行 - 设计方案

## 1. 问题描述

### 1.1 现象
工作流 id=34 中，Switcher节点"权限判断条件"条件匹配后，只执行了"临时知识库"节点，没有执行"项目管理知识库检索"节点。

### 1.2 根因分析

**Edge表数据（正确）**：
```
sourceHandle: pZkTvFMANeJ6mCvJ66dsACLrlgGxqgod
  → 临时知识库 (QWV9vpC10Lt7dL5Ilh2sgVDTwls2IJFp)
  → 项目管理知识库检索 (AbBFxq_lUJYT7OZohGI1UFzYN2mIkGbq)
```

**node_config数据（只存了一个）**：
```json
{
  "cases": [{
    "uuid": "pZkTvFMANeJ6mCvJ66dsACLrlgGxqgod",
    "target_node_uuid": "QWV9vpC10Lt7dL5Ilh2sgVDTwls2IJFp"  // 只有临时知识库
  }]
}
```

**代码逻辑问题**：
- SwitcherNode 从 node_config 读取 target_node_uuid（单值）
- 返回单个 nextNodeUuid
- 忽略了 edge 表中同一 sourceHandle 的其他目标

### 1.3 技术约束

经调研确认：
- `addConditionalEdges` 的 router 函数只能返回**单个目标**（String）
- 普通 `addEdge` 支持同源多目标自动并行执行（WorkflowEngine.java 行787-789注释）

## 2. 设计方案：虚拟并行分发节点

### 2.1 核心思路

引入"虚拟并行分发节点"（ParallelDispatchNode）作为桥接：
- 对于同一 sourceHandle 有多个目标的情况，创建虚拟节点
- addConditionalEdges: Switcher → 虚拟节点（符合单目标API）
- addEdge: 虚拟节点 → 多个目标（利用自动并行特性）

### 2.2 架构图

```
【变更前 - 只执行一个目标】
Switcher --条件路由(nextNodeUuid)--> 临时知识库
                                   ✗ 项目管理知识库检索（被忽略）

【变更后 - 多目标并行执行】
                                    ┌--> 临时知识库
Switcher --条件路由--> 虚拟并行节点 --|
                                    └--> 项目管理知识库检索
```

### 2.3 虚拟节点特性

ParallelDispatchNode 是透传节点：
- 不执行任何业务逻辑
- 只转发上游输入到下游
- 多条出边自动触发并行执行

## 3. 修改清单

### 3.1 新增 ParallelDispatchNode.java

```java
/**
 * 虚拟并行分发节点
 *
 * 用途：作为条件分支多目标的桥接节点
 * - 不执行任何业务逻辑
 * - 透传上游输入到下游
 * - 多条出边自动触发并行执行
 */
public class ParallelDispatchNode extends AbstractWfNode {

    @Override
    protected NodeProcessResult onProcess() {
        // 透传输入到输出，不做任何处理
        return NodeProcessResult.builder()
            .content(changeInputsToOutputs(state.getInputs()))
            .build();
    }
}
```

### 3.2 修改 SwitcherNode.java

返回匹配的 sourceHandle 而非 targetNodeUuid：

```java
// 变更前
nextNode = switcherCase.getTargetNodeUuid();
return NodeProcessResult.builder().nextNodeUuid(nextNode)...

// 变更后
matchedSourceHandle = switcherCase.getUuid();  // case.uuid 即 sourceHandle
return NodeProcessResult.builder().nextSourceHandle(matchedSourceHandle)...

// 无匹配时
return NodeProcessResult.builder().nextSourceHandle("default_handle")...
```

### 3.3 修改 NodeProcessResult.java

新增字段：

```java
/**
 * 条件分支匹配的 sourceHandle
 * 对应 SwitcherCase.uuid 或 "default_handle"
 */
private String nextSourceHandle;
```

### 3.4 修改 WorkflowEngine.java

#### 3.4.1 runNode() 方法
写入 nextSourceHandle 到 state：

```java
if (StringUtils.isNotBlank(result.getNextSourceHandle())) {
    returnMap.put("next_source_handle", result.getNextSourceHandle());
}
```

#### 3.4.2 processConditionalEdges() 方法

重构逻辑：
1. 按 sourceHandle 分组边
2. 单目标 sourceHandle：直接用 addConditionalEdges
3. 多目标 sourceHandle：创建虚拟节点，用普通 addEdge

```java
private void processConditionalEdges(StateGraph stateGraph) throws GraphStateException {
    // 1. 找出所有条件分支节点
    Set<String> conditionalNodeUuids = findConditionalNodeUuids();

    for (String switcherUuid : conditionalNodeUuids) {
        // 2. 获取该 Switcher 的所有出边，按 sourceHandle 分组
        Map<String, List<AiWorkflowEdgeEntity>> edgesByHandle =
            groupEdgesBySourceHandle(switcherUuid);

        // 3. 构建条件路由映射
        Map<String, String> mappings = new HashMap<>();

        for (Map.Entry<String, List<AiWorkflowEdgeEntity>> entry : edgesByHandle.entrySet()) {
            String sourceHandle = entry.getKey();
            List<AiWorkflowEdgeEntity> edges = entry.getValue();

            if (edges.size() == 1) {
                // 单目标：直接映射到目标节点
                mappings.put(sourceHandle, edges.get(0).getTargetNodeUuid());
            } else {
                // 多目标：创建虚拟并行分发节点
                String virtualNodeUuid = createParallelDispatchNode(switcherUuid, sourceHandle);
                mappings.put(sourceHandle, virtualNodeUuid);

                // 添加虚拟节点到各目标的普通边（自动并行）
                for (AiWorkflowEdgeEntity edge : edges) {
                    addEdgeToStateGraph(stateGraph, virtualNodeUuid, edge.getTargetNodeUuid());
                }
            }
        }

        // 4. 添加条件边
        stateGraph.addConditionalEdges(
            switcherUuid,
            edge_async(state -> {
                String handle = (String) state.data().get("next_source_handle");
                return handle != null ? handle : "default_handle";
            }),
            mappings
        );
    }
}
```

### 3.5 WfNodeFactory.java

注册虚拟节点类型（可选，因为虚拟节点在运行时动态创建）

## 4. 兼容性分析

| 场景 | 兼容性 | 说明 |
|------|--------|------|
| 单目标条件分支 | ✅ 完全兼容 | 无虚拟节点，直接路由 |
| 多目标条件分支 | ✅ 新增支持 | 通过虚拟节点桥接并行 |
| 现有工作流数据 | ✅ 无需迁移 | edge 表数据已正确 |
| 前端 | ✅ 无需修改 | 保存逻辑不变 |
| node_config.target_node_uuid | ✅ 忽略 | 改用 edge 表数据 |

## 5. 数据流示例

以 workflow id=34 为例：

```
1. 用户输入触发工作流

2. Switcher节点执行
   - 条件匹配成功
   - 返回 nextSourceHandle = "pZkTvFMANeJ6mCvJ66dsACLrlgGxqgod"

3. 条件路由
   - 查找 sourceHandle 对应的边
   - 发现有2条边，创建虚拟节点
   - 路由到虚拟节点

4. 虚拟节点执行
   - 透传输入
   - 2条出边自动并行

5. 并行执行
   - 临时知识库节点
   - 项目管理知识库检索节点

6. 输出合并（已有机制）
   - 各节点输出通过 node_output_{uuid} 传递
   - 下游节点可获取所有并行节点输出
```

## 7. 风险评估

| 风险 | 概率 | 影响 | 缓解措施 |
|------|------|------|----------|
| 虚拟节点增加执行开销 | 低 | 低 | 虚拟节点逻辑极简，开销可忽略 |
| 调试困难（多了虚拟节点） | 中 | 低 | 日志中标记虚拟节点，便于识别 |
