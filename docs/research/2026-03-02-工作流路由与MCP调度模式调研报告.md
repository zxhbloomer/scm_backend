# 工作流路由与MCP调度模式调研报告

> 调研日期: 2026-03-02
> 调研目标: 确定工作流自动路由和MCP工具调度的官方推荐模式
> 参考项目: Spring AI Alibaba、FastGPT、DeepResearch、LangGraph、Spring AI

---

## 一、调研背景

SCM系统当前使用 **Orchestrator-Workers** 模式做工作流路由，存在以下问题：

| 问题 | 现状 | 影响 |
|------|------|------|
| 路由prompt过重 | 3936字符，包含8个workflow + 23个MCP工具描述 | 1614 input tokens，6.5秒延迟 |
| 路由与执行耦合 | MCP工具信息混入路由决策 | 增加不必要的复杂度和token消耗 |
| 双层路由冗余 | "系统对话"workflow内部已有Classifier节点 | 重复路由，浪费资源 |

**核心问题**: 哪种路由模式才是官方推荐的？MCP工具应该在哪一层被调用？

---

## 二、各项目路由机制详细分析

### 2.1 Spring AI Alibaba（官方框架）

#### 路由模式：LlmRoutingAgent + RoutingNode

**源码位置**: `spring-ai-alibaba-agent-framework/.../agent/flow/`

**RoutingNode 的核心 Prompt**:
```
You are responsible for task routing in a graph-based AI system.

You have access to some specialized agents that can handle this task.
You can delegate the task to ONE or MULTIPLE agents for parallel execution.

The available agents and their capabilities are listed below:
- [agent_name]: [agent_description]
- [agent_name]: [agent_description]

Return a list of agent names from the list above.
Available names: [name1, name2, name3]

Example for single agent: ["prose_writer_agent"]
Example for multiple agents: ["prose_writer_agent", "code_reviewer_agent"]
```

**传入信息（仅3项）**:
1. `agent.name()` — 代理名称
2. `agent.description()` — 代理能力描述
3. `List<Message>` — 对话历史

**关键特征**:
- **不传入任何tool/MCP信息** — 路由层完全不知道工具的存在
- 支持返回多个agent并行执行
- 有重试机制（最多2次）
- 输出为JSON格式的agent名称列表

**6种官方策略**:

| 策略 | 路由成本 | 适用场景 |
|------|----------|----------|
| SEQUENTIAL | 零（固定顺序） | 流水线式处理 |
| PARALLEL | 零（全部并行） | 独立任务并行 |
| CONDITIONAL | 零（规则判断） | 确定性条件分支 |
| **ROUTING** | **1次LLM调用** | **不确定路由目标** |
| LOOP | 零（循环条件） | 迭代优化 |
| SUPERVISOR | N次LLM调用 | 复杂多轮协调 |

**结论**: Spring AI Alibaba 官方推荐 **ROUTING 策略**用于工作流自动选择——轻量级LLM路由，只传agent名称+描述。

### 2.2 FastGPT

#### 路由模式：Classifier Node + skipHandleId

**源码位置**: `FastGPT-main/packages/service/core/workflow/dispatch/ai/classifyQuestion.ts`

**Classifier Prompt**:
```
## 角色
你是一个"分类助手"，可以结合【背景知识】、【历史记录】、【上一轮分类结果】，
来判断用户当前问题属于哪一个分类，并输出分类标记。

## 背景知识
${systemPrompt}

## 上一轮分类结果
${memory}

## 分类清单
${typeList}    ← JSON格式：[{id, description}, ...]

## 分类要求
1. 分类结果必须从分类清单中选择。
2. 连续对话时，如果分类不明确，且用户未变更话题，则保持上一轮分类结果不变。
3. 存在分类冲突或模糊分类时，主语指向的分类优先级更高。

## 输出格式
只需要输出分类的 id 即可，无需输出额外内容。
```

**传入信息（仅4项）**:
1. 分类清单（id + description）
2. 背景知识（可选的系统提示）
3. 上一轮分类结果（记忆）
4. 用户输入

**skipHandleId 路由机制**:
```
用户输入 → Classifier → LLM返回分类ID
                          ↓
              匹配的分类 → edge.status = 'active'
              不匹配的   → edge.status = 'skipped'
                          ↓
              只有 active 边对应的节点被执行
```

**MCP工具位置**: 在工作流的Tool节点内部，**不参与路由决策**。

**关键特征**:
- 输出极简：只返回一个分类ID
- 有记忆机制：保留上一轮分类结果，避免话题切换时误判
- 路由成本：约100-500ms（分类prompt极短）

### 2.3 DeepResearch

#### 路由模式：Coordinator（二元LLM决策）+ State-Driven Dispatchers

**源码位置**: `deepresearch-main/src/main/java/.../node/CoordinatorNode.java`

**Coordinator 的路由逻辑**:
```java
// 唯一的LLM路由决策：检查是否触发了tool_call
if (assistantMessage.getToolCalls() != null && !assistantMessage.getToolCalls().isEmpty()) {
    nextStep = "rewrite_multi_query";   // 需要深度研究
    deepResearch = true;
} else {
    nextStep = END;                     // 直接回答
}
state.put("coordinator_next_node", nextStep);
```

**PlannerTool（路由信号工具）**:
```java
@Tool(name = "handoff_to_planner", description = "Handoff to planner agent to do plan.")
public void handoffToPlanner(String taskTitle) {
    logger.info("Handoff to planner task: {}", taskTitle);
    // 注意：无返回值，纯信号机制
}
```

**8个 Dispatcher（全部零LLM成本）**:
```java
// 每个Dispatcher都是这种极简模式：
public class CoordinatorDispatcher implements EdgeAction {
    @Override
    public String apply(OverAllState state) {
        return (String) state.value("coordinator_next_node", END);
    }
}
```

**路由成本分析**:

| 路由环节 | LLM调用次数 | 说明 |
|----------|-------------|------|
| Coordinator | 1次 | 二元判断：深度研究 or 直接回答 |
| 其他8个Dispatcher | 0次 | 纯状态读取，3-5行代码 |

**MCP工具位置**: 在Agent节点内部动态加载（ResearcherNode、CoderNode），**不在路由层**。

### 2.4 LangGraph（Python官方框架）

#### 官方推荐的4种模式

**1. Router（轻量路由）**:
```python
class Route(TypedDict):
    next: Literal["agent_1", "agent_2", "agent_3"]

route = llm.with_structured_output(Route).invoke(messages)
# 返回结构化的路由决策
```

**2. Orchestrator-Worker（重量级）**:
```python
# Orchestrator 分解任务 → Send() 分派给Workers → Synthesizer 合并
orchestrator_result = llm.with_structured_output(OrchestratorOutput).invoke(...)
for section in orchestrator_result.sections:
    send("worker", {"task": section})
```

**3. Conditional Edges（零LLM）**:
```python
graph.add_conditional_edges("node_a", routing_function, {
    "option_1": "node_b",
    "option_2": "node_c"
})
```

**4. Supervisor（循环监督）**:
```python
# 类似 Spring AI Alibaba 的 SUPERVISOR 策略
supervisor → delegate → worker → supervisor → ... → FINISH
```

**LangGraph 官方建议**:
- **简单路由用 Router** — 结构化输出，一次LLM调用
- **复杂多步任务用 Orchestrator-Worker** — 但仅在需要任务分解时
- **确定性分支用 Conditional Edges** — 零LLM成本

### 2.5 Spring AI（底层框架）

**工具调用模式**:
```java
// 方式1：ChatClient 动态绑定工具
ChatClient.create(chatModel)
    .prompt("query")
    .toolNames("tool1", "tool2")
    .call().content();

// 方式2：ToolCallAdvisor 自动循环
var chatClient = ChatClient.builder(chatModel)
    .defaultAdvisors(ToolCallAdvisor.builder()
        .toolCallingManager(toolCallingManager)
        .build())
    .build();
```

**Spring AI 没有提供路由抽象** — 路由由上层框架（Spring AI Alibaba）提供。
**工具绑定在请求级别** — 每次 ChatClient 调用可以指定不同的工具集。

## 三、核心对比

### 3.1 路由层对比

| 维度 | Spring AI Alibaba | FastGPT | DeepResearch | LangGraph | SCM现状 |
|------|-------------------|---------|--------------|-----------|---------|
| **路由方式** | LLM + agent名称/描述 | LLM + 分类ID/描述 | LLM tool_call信号 | LLM结构化输出 | LLM + 全量prompt |
| **传入信息** | agent名称+描述 | 分类ID+描述 | 系统提示+用户输入 | 选项列表 | **8 workflows + 23 MCP tools** |
| **输出格式** | agent名称列表 | 单个分类ID | 二元(tool_call/无) | 结构化Route | 任务分解计划 |
| **LLM调用次数** | 1次 | 1次 | 1次 | 1次 | **1次但token极高** |
| **预估token** | 200-500 | 150-300 | 200-400 | 100-300 | **1614** |
| **预估延迟** | 0.5-1.5s | 0.1-0.5s | 0.5-1s | 0.3-1s | **6.5s** |
| **支持并行** | 是 | 否 | 否 | 是(Send) | 否(顺序执行) |

### 3.2 MCP/Tools 位置对比

| 项目 | MCP/Tools在路由层？ | MCP/Tools在哪里？ | 路由层知道工具吗？ |
|------|---------------------|--------------------|--------------------|
| Spring AI Alibaba | **否** | Agent内部通过ChatClient绑定 | **不知道** |
| FastGPT | **否** | 工作流Tool节点内部 | **不知道** |
| DeepResearch | **否** | Agent节点内动态加载 | **不知道** |
| LangGraph | **否** | Agent/Worker内部 | **不知道** |
| **SCM现状** | **是** | 路由prompt包含23个MCP工具 | **知道全部** |

**结论一致**: 所有参考项目的路由层都**不包含MCP/Tools信息**。

## 四、官方推荐模式总结

### 4.1 路由模式推荐

```
                    用户需要什么？
                         │
              ┌──────────┼──────────┐
              ↓          ↓          ↓
         确定性路由   模糊路由    复杂多步
              │          │          │
              ↓          ↓          ↓
       Conditional   Router/     Orchestrator
         Edges     Classifier     -Worker
              │          │          │
              ↓          ↓          ↓
        零LLM成本   1次轻量LLM   N次LLM调用
```

**SCM的场景 = "模糊路由"**（用户输入→自动选择workflow），应该用 **Router/Classifier** 模式。

### 4.2 各框架的官方推荐

| 框架 | 推荐的路由方式 | 证据来源 |
|------|----------------|----------|
| **Spring AI Alibaba** | ROUTING策略（LlmRoutingAgent） | 官方6种策略中专门为此设计 |
| **FastGPT** | Classifier Node | 官方内置节点，专为分类路由设计 |
| **DeepResearch** | Coordinator + State Dispatchers | 官方示例架构 |
| **LangGraph** | Router pattern | 官方文档 workflows-agents 推荐 |

### 4.3 MCP工具调度推荐

**一致结论: MCP工具应在Workflow/Agent内部调用，不应出现在路由层。**

理由：
1. 路由的职责是"选择谁来做"，不是"用什么工具做"
2. 工具信息膨胀路由prompt，增加token和延迟
3. 工具与具体业务绑定，不应暴露给路由决策
4. Spring AI 的设计就是 per-request 绑定工具，天然支持workflow内部调用

## 五、SCM系统优化建议

### 5.1 路由层优化

**现状**: Orchestrator（全量prompt，6.5s）
**目标**: Router/Classifier（轻量prompt，< 1s）

**推荐方案: 采用Spring AI Alibaba的ROUTING策略思路**

路由prompt只需包含：
```
你是一个路由助手。根据用户输入，选择最合适的工作流。

可用工作流：
- 仓库管理助手: 仓库、库位、货位等仓储管理问题
- 项目管理知识库: 项目管理方法论、流程、制度
- 权限助手: 页面权限、按钮权限、菜单路径
- ...（只有名称+简短描述）

输出格式：只返回工作流名称，无需其他内容。
```

**预估效果**:
- Token: 1614 → ~200-300
- 延迟: 6.5s → ~0.5-1s
- 准确率: 通过keywords辅助提升

### 5.2 MCP工具层优化

**现状**: 23个MCP工具描述混在路由prompt中
**目标**: MCP工具仅在workflow内部调用

**具体做法**:
1. 路由层完全移除MCP工具信息
2. 每个workflow内部的MCP Tool节点自行管理工具
3. 如需chat级别的快捷工具调用（如"打开xxx页面"），可设置一个专门的"通用工具workflow"

### 5.3 双层路由消除

**现状**: Orchestrator路由 → "系统对话"workflow → 内部Classifier再路由
**目标**: 单层路由，直接到目标workflow

---

## 六、最终推荐

### 推荐采用的模式

**Spring AI Alibaba ROUTING策略 + FastGPT Classifier思路的结合**:

1. **路由prompt**: 只包含workflow名称 + 简短描述 + 关键词（参考FastGPT的分类清单）
2. **输出格式**: 结构化输出，返回workflow标识（参考LangGraph的structured output）
3. **MCP工具**: 全部在workflow内部调用（参考所有项目的一致做法）
4. **状态驱动**: 路由结果写入状态，后续节点读取状态执行（参考DeepResearch的Dispatcher模式）

### 不推荐的模式

1. **Orchestrator-Workers**: 当前场景不需要任务分解，是"选择"不是"分解"
2. **Supervisor循环**: 当前场景不需要多轮协调
3. **Chat级MCP调用**: 无参考项目这么做，且增加routing复杂度
